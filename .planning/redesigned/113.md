Here is a clear description for the LLM.

## Current bug

It is currently possible for the **same person/browser session** to submit the **same questionnaire multiple times**, and each submission is being counted as if it came from a **new responder**.

This is wrong.

---

# What the coordinator sees

On the coordinator **Responses** page:

* the questionnaire is in state **Open**
* the response counter shows **3 / 2** and **150%**
* that means the system has accepted **more responses than expected respondents**

The question summaries also show:

* Question 1: **Yes - 100% (3)**
* Question 2: **Option 1 - 100% (3)**
* Question 3: text responses count = **3**

In the **Responders** section, there are **three different responder IDs** listed, each with a different coloured responder marker and different IDs, even though this was produced by repeatedly responding from what should effectively be the same voter context.

So from the coordinator point of view, the app is treating repeated submissions as if they came from three distinct responders.

---

# What the voter sees

On the voter **Vote** page:

* questionnaire is visible and open
* token state says:

  * `1. Token: Token ready`
  * `2. Response ready: Token ready`
  * `3. Submitted: Response submitted`
* the bottom of the page says:

  * `Response submitted`
  * `You can close this page.`

Despite that, it is still possible to submit again in a way that ends up being counted as a new responder identity rather than being blocked, rejected, or treated as an update from the same responder.

Also visible on the voter side:

* **Participation history** exists
* but repeated submission is still not being prevented strongly enough

---

# Why this is wrong

For this course-feedback questionnaire model, one voter should only be able to contribute **one response per questionnaire**.

At minimum, the system must enforce one of these policies:

## Preferred policy

* **one responder identity per questionnaire**
* one accepted submission only
* later attempts from the same identity are rejected

## Alternative policy

* one accepted submission per responder identity
* later submissions replace the earlier one
* but they must **not** appear as a brand new responder

What is happening now is worse than both:

* repeated submits are creating effectively **new responder IDs**
* accepted count increases each time
* completion can go above 100%
* responders list grows incorrectly

---

# Likely interpretation of the bug

The app is probably generating or rotating the voter/responder identity too loosely during repeat participation.

Possible causes:

## 1. New responder marker/id generated on repeat submission

The responder identity appears to be treated as fresh on each submit or each restored participation flow, instead of staying stable for that questionnaire.

## 2. Submission uniqueness is keyed to responder marker only, not stable voter identity

If the responder marker changes, the reducer may think it is a new participant.

## 3. No server/reducer-side deduplication by questionnaire + responder identity

The system is likely accepting any valid-looking response object without checking:

* has this responder already submitted for this questionnaire?

## 4. Participation history exists in UI but is not being used as an enforcement mechanism

The voter page knows something has already been submitted, but the protocol or reducer is not treating that as final.

---

# Required invariant

The LLM should apply this strictly:

## One-response rule

For a given questionnaire:

* one stable responder identity
* one accepted response maximum

A repeated attempt from the same voter must result in exactly one of:

### Option A - reject

* do not accept second response
* do not increment counts
* do not add a new responder entry

### Option B - replace

* replace prior response for the same responder identity
* keep total responder count unchanged
* keep completion count unchanged
* keep one responder row only

But **never**:

* create a new responder identity
* increment accepted count beyond the number of real respondents
* allow completion to exceed 100%

---

# What the LLM should inspect

## Voter-side identity generation

Inspect where the responder marker / responder ID is created.

Check whether it changes:

* on refresh
* on resubmit
* on restored questionnaire flow
* on token refresh
* on each participation attempt

For one questionnaire, the responder identity should remain stable.

## Submission reducer / acceptance logic

Inspect where accepted responses are counted.

The reducer must key accepted responses by something like:

* `questionnaire_id + responder_id`

or stronger:

* `questionnaire_id + stable voter participation identity`

It must not count every new submission event as a distinct responder if the logical responder is the same.

## Responders list generation

Inspect how the Responders section is built.

It should be based on **unique logical responders**, not raw submission events.

## Completion calculation

Inspect how `n / m` and `%` are computed.

They should be based on:

* unique accepted responders
  not:
* total accepted submission events

---

# Expected correct behaviour

If the same voter submits multiple times to the same questionnaire:

## If using reject policy

* second submission blocked or rejected
* response count stays the same
* responder list stays the same
* completion stays at correct percentage

## If using replace policy

* old response replaced
* responder count stays the same
* results recomputed from latest accepted response only
* no extra responder marker/id appears

## Implementation status (2026-04-15)

Completed in `v1.00`.

- Voter response publishing now uses a stable per-questionnaire responder key per voter profile (no new responder identity per resubmit).
- Voter UI now enforces one accepted submission per questionnaire locally:
