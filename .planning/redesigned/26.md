Yes. Stop treating it as a single pass/fail harness and turn it into a **timeline harness**.

Right now you need two things:

* **stage timings**
* **per-actor state snapshots**

So design the tests to answer:

* what happened
* when
* who saw it
* what never happened

# Goal

Produce a run artifact that lets you read the protocol like this:

* coordinator app ready `+0.4s`
* questionnaire published `+1.2s`
* voter 1 saw questionnaire `+2.0s`
* voter 3 requested token `+3.1s`
* coordinator issued blind token to voter 3 `+4.4s`
* voter 3 published response `+7.2s`
* coordinator accepted response `+8.0s`
* result summary published `+10.5s`

That is what you need now.

---

# Test design

Use **three layers** of telemetry:

## 1. Global stage timeline

One ordered list of major events across the whole run.

## 2. Per-actor timelines

One timeline per:

* coordinator
* each voter

## 3. Snapshot-at-timeout

If the run fails, dump the current state for all actors.

# Global timeline events

Every event should have:

```json id="h3jd44"
{
  "tMs": 1520,
  "actor": "coord1",
  "kind": "questionnaire_published",
  "details": {
    "questionnaireId": "course_feedback_2026_term1",
    "eventId": "..."
  }
}
```

Where:

* `tMs` is milliseconds since run start
* `actor` is `coord1`, `voter1`, etc.
* `kind` is a stable event name
* `details` is small structured metadata

For each voter:
