## Concrete fix

Patch the coordinator startup logic so that **single-coordinator mode never waits for MLS join**.

### In `SimpleCoordinatorApp.tsx`

Add one canonical flag near startup:

```ts
const singleCoordinatorMode = coordinatorCount === 1;
```

Then define effective readiness:

```ts
const effectiveJoinedGroup =
  singleCoordinatorMode ? true : engineStatus.joined_group;

const effectiveGroupReady =
  singleCoordinatorMode ? true : engineStatus.group_ready;
```

And if you track phase:

```ts
const effectivePhase =
  singleCoordinatorMode && engineStatus.readiness === "waiting_for_group_ready"
    ? "ready_for_round_setup"
    : coordinatorPhase;
```

### Replace blocking checks

Anywhere you currently have checks like:

```ts
if (!engineStatus.joined_group || !engineStatus.group_ready) {
  setCoordinatorPhase("mls_join_pending");
  return;
}
```

change them to:

```ts
if (!effectiveJoinedGroup || !effectiveGroupReady) {
  setCoordinatorPhase("mls_join_pending");
  return;
}
```

### Skip startup observation/fetch loop in single-coordinator mode

If there is an effect or interval that repeatedly fetches control carriers while startup is pending, wrap it:

```ts
if (singleCoordinatorMode) {
  return;
}
```

This should apply to:

* startup control-carrier fetch loop
* forced startup recovery loop
* startup publish probe loop

Because for one coordinator there is no control-group observation to wait for.
