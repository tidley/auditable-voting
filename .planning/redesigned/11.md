That is a decisive result.

You now have direct evidence that the problem is **not** generic relay failure and **not** generic transport collapse.

## What Phase 10 proved

For the probed row:

* the published ticket event **exists**
* it is retrievable from a write-success relay with **`kind_only`**
* it is **not** returned with:

  * `kind + mailbox`
  * `kind + mailbox + etype`
* the live/backfill **mailbox ids do not match** the request mailbox id for that row

So the next decision is now clear:

> **The primary blocker is mailbox-id / mailbox-filter mismatch, not “relays are dropping the event”.**

That is exactly the kind of concrete decision-point evidence you wanted.

# What this means

You can now rule out a lot of dead ends:

## Not the main problem

* Rust acceptance
* coordinator mapping
* scheduler alone
* resend alone
* “relays never stored the event”
* “publish success is always fake”

## Main problem

The writer and reader are **not aligned on mailbox addressing/query shape**.

At least one of these is true:

1. publish path uses one mailbox id, read path derives another
2. publish path writes tags the reader does not later query correctly
3. reader is over-constraining by mailbox tag and/or `etype`
4. request mailbox id is being reused incorrectly for ticket fetch, or vice versa
5. some rows have multiple mailbox ids in flight and the wrong one is chosen at read time

Because `kind_only` finds the event, but `kind + mailbox` does not, the problem is almost certainly in the **mailbox tag semantics or mailbox-id derivation symmetry**.

# Next phases

## Phase 11 - Fix mailbox-id symmetry first

## Goal

Guarantee that the ticket write path and ticket read path use the **same mailbox id** for the same voter and round.

## Required work

### 11.1 Make one canonical mailbox-id function

Create exactly one function for mailbox derivation and use it everywhere.

It should take only:

* `election_id`
* `round_id`
* voter identifier
* mailbox salt or deterministic mailbox input

Do **not** allow multiple ad hoc derivations.

### 11.2 Add explicit publish-vs-read mailbox comparison in-app

For each voter row, store and expose:

* `requestMailboxId`
* `ticketPublishMailboxId`
* `ticketReadMailboxId`
* `ticketBackfillMailboxId`

Add a derived flag:

* `mailboxIdConsistent`

### 11.3 Fail loudly on mismatch

If:

* `ticketPublishMailboxId !== ticketReadMailboxId`
  or
* `ticketPublishMailboxId !== ticketBackfillMailboxId`

then:

* emit explicit diagnostic
* do not silently continue as if transport is fine

## Exit condition

For every row, publish/read/backfill mailbox ids are identical.

## Phase 12 - Remove mailbox tag over-constraint temporarily

## Goal

Confirm whether mailbox-tag filtering is the only thing preventing observation.

## Why

You already proved `kind_only` works while `kind + mailbox` does not. That means the narrow filter is currently not trustworthy.

## Required work

### 12.1 Temporary diagnostic fallback

For ticket observation/backfill only, add a fallback mode:

1. try `kind + mailbox`
2. if zero results and publish success exists, retry with:

   * `kind_only`
   * optionally `kind + election + round`
3. reconcile locally by:

   * `request_id`
   * `ticket_id`
   * expected recipient identity
   * expected round

This is a diagnostic and fallback path, not necessarily the final permanent design.

### 12.2 Drop `etype` from critical path

Since `kind + mailbox + etype` also failed in the probe, remove `etype` from the critical observation/backfill filter for now.

Use it only as:

* optional narrowing when already known-good
* or local parsed validation after retrieval

## Exit condition

Previously missing tickets can now be seen via the broader fetch path.

## Phase 13 - Separate request mailbox from ticket mailbox if needed

## Goal

Determine whether using the same mailbox namespace for request and ticket is the source of the mismatch.

## Why

Your report specifically says the live/backfill mailbox ids did not match the **request mailbox id** for the probed row. That raises the possibility that request and ticket flows are deriving or expecting different mailbox ids without that being made explicit.

## Required work

Choose one of these and make it explicit:

### Option A - one mailbox per voter per round

Use one mailbox id for:

* request
* ticket
* ack

### Option B - distinct mailbox ids per object class

Use:

* `request_mailbox_id`
* `ticket_mailbox_id`
* `ack_mailbox_id`

If you choose B, then:

* do not compare ticket fetch mailbox against request mailbox anymore
* expose all three explicitly
* use the correct one per event class

For the initial deployment, I would strongly favour **Option A** unless there is a compelling reason otherwise.

## Exit condition

There is no ambiguity about which mailbox id belongs to which event class.
