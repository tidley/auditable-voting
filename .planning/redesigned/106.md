## Coordinator navigation

There should be **one tab row only** in coordinator:

* **Build**
* **Responses**
* **Settings**

Remove the extra layer of navigation entirely.
No second row with things like:

* Audience
* Publish
* Results
* similar lifecycle/admin tabs

Those concepts should be folded into the three top-level tabs.

---

# 1. Build tab

This is the default tab and the main authoring screen.

It should contain **all questionnaire creation and lifecycle controls** in one clear vertical page.

## Build tab contents

### A. Questionnaire identity

Top section.

Fields:

* **Questionnaire ID**

  * auto-generated
  * copy button
  * show QR button if useful
* **Name**
* **Description**

Rules:

* no default questions
* no default filler text beyond placeholder text
* ID is shown but not treated as the main focus

---

### B. Questionnaire state

A compact status section near the top.

Show:

* **State**

  * Draft
  * Saved
  * Published
  * Open
  * Ended
  * Counted
* **Opened**

  * date/time if opened
* **Closing / Closed**

  * date/time if configured or closed
* **Responses**

  * `n / m`
  * percentage
* **Coordinators**

  * e.g. `1 of 1`
  * can say `todo` or hidden if not yet supported

Rules:

* this is informational first
* do not scatter lifecycle buttons elsewhere
* status should be visible at a glance

---

### C. Questions

Main editor section.

Show:

* section title: **Questions**
* button row:

  * `Add yes/no question`
  * `Add multiple choice question`
  * `Add text question`

No default questions at first load.

Each question must be shown as a **clearly separated card**.

Each question card should include:

* question number
* question type
* prompt field
* required toggle
* type-specific options
* actions:

  * duplicate
  * move up
  * move down
  * delete

Spacing must be generous so one question does not visually merge into the next.

---

### D. Readiness checklist

Below the questions.

Show checklist items such as:

* Title added
* Description added
* At least one question added
* Respondents known
* Questionnaire not yet published

This should make it obvious what is missing before publish.

---

### E. Publish questionnaire button

At the bottom of Build.

Single clear primary button:

* **Publish Questionnaire**

Rules:

* disabled until checklist requirements are met
* no separate Publish/Open/Close buttons all sitting together in the middle of the page
* once published, button changes according to state:

  * `Open Questionnaire`
  * `Close Questionnaire`
  * or becomes informational if already ended

But this lifecycle control still stays in **Build**, not in a separate Publish tab.

---

# 2. Responses tab

This is only for monitoring responses.

## Responses tab contents

Show:

* accepted responses count
* rejected responses count
* latest response activity
* percentage complete
* simple list or summary of respondents if needed
* state of questionnaire:

  * open / closed / counted

This tab should answer:

> Are people responding, and how many?

Rules:

* no question editing here
* no publish controls here
* no coordinator management here
* no clutter

If results later exist, they can still be shown here as response summaries if you want, but not as a separate top-level tab.

---

# 3. Settings tab

This is for secondary configuration and lower-priority admin.

## Settings tab contents

Put here:

* coordinator management
* additional coordinators / sub-coordinators if supported
* advanced diagnostics
* relay/debug/protocol details
* any dangerous or uncommon actions
* optional QR settings if not shown in Build
* export/import if needed

This tab should answer:

> What advanced configuration or administration is available?

Rules:

* this should not contain the main authoring workflow
* this should not be open by default
* debug/protocol information belongs here, not in Build

---

# 4. Strict layout rule

## Only one visible navigation row

At the top of the coordinator screen:

* Build
* Responses
* Settings

Nothing underneath as a second menu row.

No nested page navigation like:

* Audience
* Publish
* Results

Those functions must be absorbed into the three-tab structure above.

---

# 5. Mapping from old concepts to new tabs

To make this strict:

* **Audience** -> move into **Build** only as checklist/respondent-known summary, or into **Settings** if advanced
* **Publish** -> move into **Build**
* **Results** -> move into **Responses**
* **Coordination / coordinator management** -> move into **Settings**
* **Debug / protocol state** -> move into **Settings**

---

# 6. Behaviour rules

## On first load

Open **Build**.

Show:

* empty questionnaire
* no default questions
* clear “Add question” actions
* publish disabled until ready

## On published/open questionnaire

Still use the same three tabs:

* Build = questionnaire definition + lifecycle state
* Responses = live response monitoring
* Settings = advanced/admin

---

# 7. Minimal page summary

## Build

Create questionnaire, define questions, see status, publish/open/close.

## Responses

See response progress and outcome summaries.

## Settings

Manage coordinators and advanced/technical options.

---

# 8. Strong recommendation

The coordinator UI should feel like:

> one questionnaire builder with two supporting tabs

not:

> an admin console with many similar sections and overlapping menu layers

Your hand-drawn structure is the correct direction.
