# Core failure pattern

The system is stuck in this contradictory state:

## Coordinator side

* questionnaire exists
* questionnaire published
* metadata visible
* questions partially degraded after tab switch
* zero responses

## Voter side

* questionnaire visible
* token ready
* questions answerable locally
* but still “waiting for the next live round and ticket”
* cannot submit

So the system has split into **three inconsistent truths**:

1. **Coordinator metadata truth**

   * questionnaire is published/opened

2. **Voter lifecycle truth**

   * still effectively draft / not live for submission

3. **Questionnaire content truth**

   * partially lost / degraded when coordinator switches tabs

---

# Likely interpretation for the LLM

The bug is probably one or more of these:

## 1. Questionnaire definition is not being persisted/restored correctly across tabs

Symptoms:

* full questionnaire exists when authored
* coordinator Responses tab only shows degraded question text
* likely only partial summary survives instead of canonical question objects

The LLM should inspect:

* where questionnaire draft/build state is stored
* where response/result tab reloads questionnaire state from
* whether Responses tab uses a reduced derived summary instead of canonical question definitions

## 2. Published/open state is not propagating consistently to voter-facing live state

Symptoms:

* coordinator sees `Published`
* voter sees `State: Draft`
* voters cannot get a live ticket

The LLM should inspect:

* whether `publish questionnaire` updates only coordinator-local state
* whether voter-facing round/open/live state is driven by a different reducer or event source
* whether there is a missing `open` transition or missing public publication after publish

## 3. Ticket issuance / live-round formation is not happening even though the questionnaire is visible

Symptoms:

* voter has token ready
* questionnaire loaded
* but `No live vote ticket yet`
* `Waiting for the next live round and ticket`

The LLM should inspect:

* whether publish is insufficient and open is still required
* whether the single-questionnaire/single-round path is incorrectly still waiting for a multi-round ticket lifecycle
* whether voter live-ticket retrieval is keyed off a round-open event that was never emitted or never observed

## 4. Submit button enablement is tied to live ticket state, not questionnaire visibility

Symptoms:

* answers are filled
* token ready
* response ready
* but submit still disabled

The LLM should inspect:

* exact predicate for enabling `Submit response`
* likely it requires `liveTicket != null`
* determine why liveTicket is not materialising

---

# What the LLM should treat as the target fix

The desired correct behaviour is:

## After questionnaire is published/opened

* voters see the same questionnaire content as coordinator authored
* voters see correct lifecycle state
* a live ticket becomes available
* `Submit response` becomes enabled once token + live ticket + answers are ready

## After coordinator switches tabs

* questions remain fully intact
* Responses tab should show the same prompts/options as Build
* not degraded placeholders or truncated strings
