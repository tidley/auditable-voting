That is a serious correctness bug.

## What is happening

A brand new voter, in a different browser, can already see the questionnaire before the coordinator has published it.

From the screenshot:

### Coordinator side

* coordinator is on **Build**
* questionnaire state is clearly:

  * **Draft**
* there are no questions yet in the builder section
* this is still an unpublished draft questionnaire

### Voter side

* voter is on **Vote**
* voter can already see:

  * questionnaire title/info
  * state shown as **Open**
  * token status
  * a visible question card
  * an enabled-looking questionnaire flow

So the voter is seeing questionnaire content that should still be private to the coordinator draft.

# Why this is wrong

A voter must **never** see a questionnaire until it has been explicitly published.

More specifically:

## Draft questionnaires must be coordinator-private

While in `Draft`:

* visible only to the coordinator
* not discoverable by voters
* not returned by voter queries
* not treated as active/open
* not selectable in voter UI

## Voter visibility should begin only after publish

The earliest a voter should see a questionnaire is after:

* questionnaire is published
* and, depending on lifecycle, possibly opened

---

# Likely bug classes

## 1. Voter discovery is including draft records

The voter discovery path is probably querying:

* all questionnaire definitions
* or all locally/restored questionnaire-like objects
* without filtering by lifecycle state

That would let draft questionnaires leak into voter view.

## 2. Draft data is being written to a public/discoverable channel too early

The coordinator may be publishing questionnaire definition content before “Publish Questionnaire” is clicked.

For example:

* autosave or draft save may be using the same event kind/path as publish
* build-time state may already be hitting relays
* voter then discovers that draft object

## 3. Voter UI is trusting questionnaire metadata without checking publish state

Even if a draft questionnaire is somehow visible, the voter UI must still reject it unless:

* state is `published` or `open`
* and it passes active/relevant filtering

## 4. State mapping is wrong

The voter page is showing `State: Open` while the coordinator page shows `State: Draft`.

So there may be:

* mismatched state derivation
* fallback default to `Open`
* stale old public state mixed with current draft content

---

# Required acceptance rule

The LLM should apply this strictly:

## Draft visibility rule

If questionnaire state is:

* `draft`
* `saved`

then:

* it must not appear in voter questionnaire discovery
* it must not appear in voter dropdowns
* it must not render on the voter vote page
* it must not be treated as active
* it must not be queryable except by explicit coordinator-private paths

## Only these states may be voter-visible

A questionnaire may be visible to voters only if state is:

* `published`
* `open`

And even then it must pass:

* not expired
* relevant coordinator context
* not archived/stale

---

# What the LLM should inspect

## Coordinator side

Check whether build/draft actions are writing to:

* public questionnaire definition events
* relay-published objects
* globally discoverable local caches

before publish is clicked.

## Voter side

Check whether voter discovery/filtering is missing:

* `state in (published, open)` filter
* `not draft`
* `not saved`
* `not closed`
* coordinator-context filtering

## Shared reducer/state layer

Check whether:

* draft state and published state use the same object path
* draft content can overwrite or masquerade as published content
* default lifecycle fallback incorrectly becomes `open`

# Correct behaviour

## Coordinator in Draft

* coordinator can build questionnaire
* questionnaire exists only locally/coordinator-private
* voter sees:

  * `No active questionnaire yet`

## After Publish

* questionnaire becomes discoverable to voters
* voter may then see it in dropdown/current questionnaire card

## Before Publish

* no question prompt
* no token state
* no answer controls
* no “State: Open”
* no submit path

# Short bug report for the LLM

A brand new voter in a different browser can see and interact with a questionnaire that is still in **Draft** on the coordinator side.

This is a correctness bug.

### Expected

Draft questionnaires are coordinator-private and must not appear in voter discovery or render on the voter vote page.

### Actual

Voter page shows questionnaire content and `State: Open` even though coordinator Build page still shows `State: Draft`.

### Required fix

Strictly enforce that voter-visible questionnaires must be filtered to:

* `published`
* `open`

and draft/saved questionnaires must never be surfaced to voter discovery or vote rendering.

## Implementation status (2026-04-14)

Completed in `v0.96`.

- Voter questionnaire discovery no longer infers lifecycle from definition timestamps when no explicit state exists.
- Voter discovery now treats missing explicit state as `draft`, so draft definitions are not discoverable.
- Restore-by-ID now requires explicit voter-visible lifecycle (`published` or `open`) and rejects unpublished drafts.
- Voter render refresh now hides questionnaire content unless explicit state is `published` or `open`.
