Below is a detailed protocol for the **simplified private-first course-feedback system**:

* **1 coordinator**
* **many voters**
* **1 questionnaire**
* **1 response bundle per voter**
* **ephemeral response key per submission**
* **public questionnaire**
* **private answers**
* **public aggregate results**
* **no ticket/mailbox/ack choreography**
* **no MLS runtime dependency in single-coordinator mode**

This is the version that keeps privacy reasonably good while removing a very large amount of transport and coordination complexity.

---

# 1. Protocol goals

## Must achieve

* public distribution of the questionnaire
* private submission of answers
* pseudonymous response identity
* replayable public state
* one response bundle per voter
* browser-native operation
* simple enough to ship

## Explicit non-goals for v1

* threshold decryption
* blind-token unlinkable eligibility
* multi-coordinator consensus
* fully public cryptographic verifiability of admission
* perfect anonymity against a powerful network observer

---

# 2. Core model

There are only **three object classes**:

1. **Questionnaire definition**
   Public. Everyone sees the same questions.

2. **Encrypted response bundle**
   Private. Each voter submits one encrypted answer bundle.

3. **Aggregate result summary**
   Public. Coordinator publishes counts/summaries after close.

That is the whole protocol.

---

# 3. Actors

## Coordinator

The organiser of the questionnaire.

Responsibilities:

* define and publish questionnaire
* publish open/close state
* receive encrypted responses
* decrypt and validate responses
* tally results
* publish aggregate results

## Voter

A participant submitting one response bundle.

Responsibilities:

* fetch questionnaire
* generate ephemeral submission key
* fill answers
* encrypt response bundle
* publish response event

## Relays

Used for:

* public questionnaire distribution
* encrypted response event distribution
* result publication

Relays are not trusted for confidentiality.

---

# 4. Identity model

## Coordinator identity

The coordinator has a long-term Nostr pubkey:

* `coordinator_pubkey`

This key is used for:

* publishing questionnaire definition
* publishing result summary
* publishing coordinator metadata

The coordinator also has an **encryption public key**.
For simplicity in v1, this may be the same logical identity if the crypto library supports encryption from the same key material, or a dedicated coordinator encryption key can be published in the questionnaire definition.

## Voter identity

Each voter may have:

* a normal long-term identity, or
* no long-term identity requirement at all, depending on policy

But the actual response submission should be signed by a **fresh ephemeral keypair**:

* `response_pubkey`
* `response_privkey`

This keypair is generated per response submission.

That means the response is not directly signed by the voter’s usual key.

---

# 5. Visibility / privacy model

## Public

Publicly visible:

* questionnaire definition
* questionnaire metadata
* open/close state
* result summary
* response event existence
* response ephemeral pubkey
* ciphertext blob / encrypted response object

## Private

Private:

* actual response contents
* especially free-text answers

## Pseudonymity

Each response is signed by a fresh ephemeral key, so:

* responses are not trivially linked to a long-term user identity
* responses from the same person across different questionnaires can be separated

This gives **pseudonymity**, not perfect anonymity.

## 6.1 Questionnaire definition

### Event type

`questionnaire_definition`

### Visibility

Public

### Purpose

Defines the full questionnaire and policy.

### Required fields

```json id="n6e6p0"
{
  "schema_version": 1,
  "event_type": "questionnaire_definition",
  "questionnaire_id": "course_feedback_2026_term1",
  "title": "Course feedback",
  "description": "Please answer all required questions.",
  "course_name": "Module ABC",
  "created_at": 1712530000,
  "open_at": 1712533600,
  "close_at": 1712619999,
  "coordinator_pubkey": "npub1coordinator...",
  "coordinator_encryption_pubkey": "hex_or_npub",
  "response_visibility": "private",
  "eligibility_mode": "open",
  "allow_multiple_responses_per_pubkey": false,
  "questions": [
    {
      "question_id": "q1",
      "type": "yes_no",
      "prompt": "Was the course material clear?",
      "required": true
    },
    {
      "question_id": "q2",
      "type": "multiple_choice",
      "prompt": "How would you rate the pace?",
      "required": true,
      "multi_select": false,
      "options": [
        { "option_id": "slow", "label": "Too slow" },
        { "option_id": "good", "label": "About right" },
        { "option_id": "fast", "label": "Too fast" }
      ]
    },
    {
      "question_id": "q3",
      "type": "free_text",
      "prompt": "What should be improved?",
      "required": false,
      "max_length": 1000
    }
  ]
}
```

### Validation rules

* `questionnaire_id` unique
* `coordinator_pubkey` present
* `coordinator_encryption_pubkey` present
* every `question_id` unique
* each multiple-choice question has at least 2 options
* `option_id` unique within each question
* `open_at < close_at`

* `open`
* `closed`
* `results_published`

### Rule

## 6.3 Encrypted response bundle

### Event type

`questionnaire_response_private`

### Visibility

Public event, private payload

### Purpose

Carries one encrypted response bundle from one ephemeral submission key.

### Required fields

```json id="x8l5mq"
{
  "schema_version": 1,
  "event_type": "questionnaire_response_private",
  "questionnaire_id": "course_feedback_2026_term1",
  "response_id": "resp_7f3f5e2a",
  "created_at": 1712537200,
  "author_pubkey": "npub1ephemeral...",
  "ciphertext_scheme": "x25519_xchacha20poly1305_v1",
  "ciphertext_recipient": "coordinator_encryption_pubkey",
  "ciphertext": "base64...",
  "payload_hash": "hex_sha256"
}
```

### Important note

The event itself is public, but the `ciphertext` hides the answer contents.

### Response identity

The event is signed by the **ephemeral response key**.

## 6.4 Decrypted response payload

This is the plaintext inside the encrypted bundle.

```json id="s9v1fw"
{
  "schema_version": 1,
  "kind": "questionnaire_response_payload",
  "questionnaire_id": "course_feedback_2026_term1",
  "response_id": "resp_7f3f5e2a",
  "submitted_at": 1712537200,
  "answers": [
    {
      "question_id": "q1",
      "answer_type": "yes_no",
      "value": true
    },
    {
      "question_id": "q2",
      "answer_type": "multiple_choice",
      "selected_option_ids": ["good"]
    },
    {
      "question_id": "q3",
      "answer_type": "free_text",
      "text": "More worked examples would help."
    }
  ]
}
```

### Validation rules

* `questionnaire_id` must match outer event
* every `question_id` must exist in questionnaire definition
* answer type must match declared question type
* all required questions must be answered
* all selected option ids must be valid for that question
* free text length must be within limits

# 7. Submission flow

## Step 1 - coordinator publishes questionnaire

Coordinator publishes:

* `questionnaire_definition`
* later `questionnaire_state(open)`

## Step 2 - voter fetches questionnaire

Voter app subscribes to:

* `questionnaire_definition`
* `questionnaire_state`

for the selected questionnaire id.

## Step 3 - voter fills form

UI renders all questions.
The voter fills in one form.

## Step 4 - voter generates ephemeral response keypair

Before submit:

* generate fresh keypair
* use this keypair to sign the response event

## Step 5 - voter builds response payload

Assemble:

* `response_id`
* `questionnaire_id`
* typed `answers[]`

## Step 6 - voter encrypts response payload

Encrypt plaintext response payload to:

* `coordinator_encryption_pubkey`

## Step 7 - voter publishes encrypted response event

Publish `questionnaire_response_private` to relays.

## Step 8 - coordinator fetches encrypted responses

Coordinator subscribes to:

* `questionnaire_response_private` for the questionnaire id

## Step 9 - coordinator decrypts and validates

For each response:

* decrypt
* validate against questionnaire definition
* accept or reject

## Step 10 - coordinator tallies after close

After questionnaire closes:

* aggregate quantitative answers
* prepare free-text export privately
* publish `questionnaire_result_summary`

---

# 8. Acceptance rules

For v1, keep this simple.

## Default mode: open submission

Accept the **first valid response per ephemeral pubkey**.

This is weak against Sybil duplicates, but simple.

## Better mode: allowlist by participant identity

If you need some control, add an allowlist of long-term pubkeys or participant ids.

Then require the response payload to include an **eligibility claim** signed by the long-term key before encryption.

But this adds some complexity and weakens anonymity.

## Recommended v1

If this is a real course-feedback pilot and not adversarial public voting:

* use organisational process controls
* accept one response event per ephemeral pubkey
* optionally rate-limit or moderate coordinator-side duplicates

The protocol stays much simpler.

# 9. UI mode switch

## Coordinator UI top-level toggle

Add:

* `Response visibility`

  * `Public`
  * `Private`

For now, default to:

* `Private`

## Behaviour

### Public mode

* response event plaintext
* easy debugging
* no answer encryption

### Private mode

* response event ciphertext
* questionnaire still public
* answer bundle encrypted

## If only one mode is shipped now

Ship:

* `Private`

because that is what you said you prefer, and it is still much simpler than the old ticket/mailbox design.

# 10. Encryption model

## Recommended

Encrypt the full response payload to the coordinator encryption public key.

## Do not do

* shared static frontend secret
* fake “one-way key”
* XOR/obfuscation
* ad hoc reversible encoding

## Practical cryptographic shape

Use a standard asymmetric envelope encryption scheme, for example conceptually:

* voter generates ephemeral symmetric content key
* encrypt payload with symmetric authenticated encryption
* encrypt content key to coordinator public key
* publish ciphertext bundle

The exact library choice can vary, but the protocol assumption is:

* only coordinator can decrypt
* relays cannot read answers
* anyone can store/forward the ciphertext

# 12. Reducer / validation rules

## Public reducer

Tracks:

* questionnaire definition
* open/closed state
* result summary

## Private response acceptance reducer

Coordinator-side only.

For each incoming response:

1. decrypt
2. verify payload hash if used
3. validate schema
4. check questionnaire open
5. check duplicate rule
6. accept or reject

## Rejection reasons

Use explicit reasons:

* `questionnaire_closed`
* `unknown_question_id`
* `missing_required_answer`
* `invalid_option_id`
* `free_text_too_long`
* `duplicate_response`
* `decryption_failed`
* `invalid_payload_shape`

# 13. Public vs private switch semantics

## If UI mode = Public

Use:

* `questionnaire_response_public`
* plaintext `answers[]`

## If UI mode = Private

Use:

* `questionnaire_response_private`
* encrypted payload

Everything else stays mostly the same:

* same questionnaire definition
* same validator
* same result summary shape

That means the UI switch is practical and not too invasive.

---

# 14. What is lost relative to the earlier architecture

You lose:

* unlinkable blind-token eligibility
* multi-coordinator trust distribution
* threshold issuance/decryption
* stronger one-person-one-vote cryptographic guarantees

You keep:

* public question distribution
* private answers
* ephemeral response identities
* browser-native flow
* much simpler architecture

For a first private version, this is a very good trade.
