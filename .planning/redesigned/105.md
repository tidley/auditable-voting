The most likely cause is **an aggressive relay connection/reconnection loop on startup**.

From the screenshot, the page is immediately throwing repeated errors for:

* `wss://relay.snort.social/`
* `wss://relay.damus.io/`

with:

* websocket connection refused
* `503 Service Unavailable`

If your startup path is trying to connect to several relays, retrying fast, and updating UI state/logging on every failure, Firefox can lock up from:

* repeated websocket creation
* repeated failed reconnect timers
* repeated state updates / rerenders
* repeated console logging
* possibly multiple tabs/components each doing the same thing

## What to fix first

### 1. Do not connect to relays on first load unless needed

The first-load voter page should be idle.

On first open, it should only show:

* local voter id
* add coordinator input
* scan button
* no active relay traffic until a coordinator is added or a vote page is opened

### 2. Add strict relay connection backoff

For each relay:

* first retry after 2 seconds
* then 5 seconds
* then 10 seconds
* then 30 seconds
* cap there

Do not retry in a tight loop.

### 3. Ensure only one relay manager exists

Very likely problem:

* component renders
* relay client created
* render again
* another relay client created
* all retry independently

Make relay connections singleton/service-scoped, not component-scoped.

### 4. Stop logging every repeated failure

Log first failure, then suppress repeats for a window.

For example:

* log first failure immediately
* then at most once every 30 seconds per relay

### 5. Do not let Configure / Vote / Settings tabs each initialise their own subscriptions

Only the active page should subscribe, and only when needed.

### 6. Add startup safe mode

If all relays fail:

* mark relay state as unavailable
* stop reconnect storm
* show small warning:

  * `Relays unavailable. Retrying in 30s.`

Do not keep hammering.
