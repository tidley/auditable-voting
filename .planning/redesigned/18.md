Below is the rewritten **literal patch plan by file** for a **blind-signature-based questionnaire protocol** with:

* **public questionnaire definition**
* **private registration / eligibility request**
* **blind-signed one-use response token**
* **ephemeral response key**
* **one response bundle per voter**
* **optionally public or encrypted response payload**

This plan assumes the immediate target remains:

* **1 coordinator**
* **many voters**
* **1 questionnaire**
* **1 response bundle per voter**

and that you want to preserve anonymity of the voter’s choices via unlinkable credentials.

---

# Patch objective

Replace the current response model with this flow:

1. coordinator publishes **public questionnaire definition**
2. voter sends **private eligibility request**
3. coordinator returns **blind signature** over a one-use token
4. voter unblinds and obtains **anonymous one-use response token**
5. voter generates **ephemeral response key**
6. voter submits **one response bundle**
7. verifier accepts **first valid response per token nullifier**
8. coordinator publishes aggregate result summary

---

# Protocol split

## Public objects

* questionnaire definition
* questionnaire state (optional)
* result summary
* optionally public response events if you choose public payload mode

## Private / restricted objects

* eligibility request
* blind token issuance response
* optionally encrypted response payload

## Identity rules

* long-term identity may be used only for eligibility request
* actual response is signed by **ephemeral key**
* acceptance is based on **blind token nullifier**, not voter pubkey

# Patch 6 - Add blind-token verifier reducer

## File

Create:
`auditable-voting-core/src/questionnaire_state.rs`

## Goal

Implement deterministic acceptance and tally rules.

## Add accepted/rejected response structs

```rust id="7nqchh"
pub struct AcceptedBlindResponse {
    pub questionnaire_id: String,
    pub response_id: String,
    pub author_pubkey: String, // ephemeral
    pub token_nullifier: String,
    pub submitted_at: u64,
    pub answers: Option<Vec<QuestionAnswer>>,
    pub payload_hash: Option<String>,
}

pub struct RejectedBlindResponse {
    pub questionnaire_id: String,
    pub response_id: String,
    pub author_pubkey: String,
    pub token_nullifier: Option<String>,
    pub submitted_at: u64,
    pub reason: String,
}
```

## Implement acceptance rules

For each response:

1. questionnaire exists
2. submission within open window
3. token proof valid against `blind_signing_pubkey`
4. token proof bound to this `questionnaire_id`
5. `token_nullifier` has not already been accepted
6. payload shape valid
7. if payload mode = public:

   * validate answers directly
8. if payload mode = encrypted:

   * acceptance may be split:

     * cryptographic acceptance of token/proof now
     * answer validation after coordinator decrypts payload

## Duplicate rule

Replace:

* first valid response per pubkey wins

with:

* **first valid response per token nullifier wins**

## Rejection reasons

Add:

* `invalid_token_proof`
* `nullifier_already_spent`
* plus earlier schema validation reasons

## Acceptance criteria

* Rust reducer accepts first valid response per nullifier, not per pubkey

# Patch 7 - Add optional encrypted payload decrypt path

## File

## Goal

Handle encrypted answer payload mode.

## Add functions

* `decrypt_questionnaire_response_payload(...)`
* `validate_decrypted_questionnaire_answers(...)`

## Important split

If payload mode = `encrypted`:

### Public verifier can verify

* token proof valid
* nullifier unique
* event shape valid

### Coordinator-only verifier can additionally verify

* decrypted answers valid
* tally from decrypted answers

This means public verification is **partially weaker** in encrypted mode, which is expected.

## Acceptance criteria

* encrypted mode is supported without changing the blind-token admission model
