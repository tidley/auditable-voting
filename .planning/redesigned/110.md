This is a good diagnosis overall. The strongest points are:

* **debounce/dedupe persistence**
* **namespace all local browser state**
* **kill broad rerenders / timer-driven rerenders**
* **replace fixed polling with backoff**
* **reduce subscription fan-out**

Those are exactly the kinds of issues that cause Firefox freezes, tab jank, and relay storms.

## My comments on each point

### 1. Debounce IndexedDB writes

Strongly agree.

This is probably one of the highest-value fixes. Broad dependency effects writing into IndexedDB can absolutely produce lock contention, UI stalls, and retry storms.

Refinement:

* do not only debounce
* also **skip writes when the serialised payload hash is unchanged**
* and split writes by domain:

  * actor/session state
  * questionnaire state
  * relay cache
  * diagnostics

### 2. Namespace `localStorage`

Agree.

This is not glamorous, but it is important. If questionnaire IDs, restore IDs, or role-specific state share browser-global keys, local testing becomes misleading and tabs contaminate each other.

Refinement:
Use a strict key scheme:

```text id="4zqhmu"
app:auditable-voting:{mode}:{actorId}:{key}
```

Examples:

* `app:auditable-voting:voter:b93cb43:questionnaireId`
* `app:auditable-voting:coordinator:16ez7fv:restoreId`

### 3. Kill the 1-second whole-coordinator rerender

Very strong agree.

This is classic death-by-timer. A huge component with a `nowMs` tick every second will constantly invalidate memoisation and re-render unrelated UI.

Refinement:

* remove global `nowMs`
* move “time ago” displays into tiny leaf components or hooks
* update them every 15s or 30s, not every second
* preferably only when visible

### 4. Backfill polling too aggressive

Strongly agree.

The quoted 2s / 4s / 5s / 15s fixed cadence is too noisy, especially when several tabs are open. It also hides real state bugs behind relay hammering.

Refinement:
Use:

* immediate live subscription
* one initial backfill on entry
* one quick retry after first miss
* then exponential backoff
* reset backoff only on meaningful state change

### 5. Consolidate subscriptions

Agree.

One subscription per request/round/coordinator is fragile and expensive. This will especially hurt on Firefox and weaker devices.

Refinement:

* group by `(election_id, round_id, plane)`
* dedupe mailbox ids before subscribing
* centralise subscription ownership in one relay service
* UI should consume derived state, not open subscriptions directly

### 6. Cache QR generation

Agree, but medium priority.

Worth doing, but not the main blocker unless QR code generation is happening repeatedly during rerenders.

### 7. Split large apps

Strongly agree.

This is probably structurally necessary. `SimpleCoordinatorApp.tsx` and `SimpleUiApp.tsx` sound like they currently mix:

* storage
* relay orchestration
* protocol state
* diagnostics
* rendering
* form logic

That makes performance and correctness hard to reason about.

### 8. Avoid broad kind-only scans

Strongly agree.

This is both a performance problem and a correctness smell. Pulling huge event sets then filtering locally is acceptable for development, not for a stable app.

Refinement:

* always constrain by `since`
* constrain by `election_id`
* constrain by `round_id`
* constrain by `author` / `coordinator` / `mailbox` when known

### 9. Throttle debug object writes

Agree.

Debug globals are useful, but giant debug objects rebuilt frequently are a hidden performance tax.

Refinement:

* only update on meaningful state change
* or behind `debug=1`
* or throttle to every 2 to 5 seconds
