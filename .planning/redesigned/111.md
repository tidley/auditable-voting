## Voter questionnaire selector - acceptance spec

This spec is strict.

The voter UI must **not** behave like a historical questionnaire browser.

The questionnaire selector must show only questionnaires that are currently relevant to the voter.

---

# 1. Purpose

The voter questionnaire selector exists to let the voter:

* access the current active questionnaire
* optionally access a small number of still-relevant questionnaires
* manually restore a questionnaire by ID if needed

It must **not**:

* list every questionnaire ever seen
* list old local test questionnaires
* list archived or closed questionnaires by default
* mix questionnaires from unrelated coordinators or actor sessions

---

# 2. Main rule

## Required behaviour

The main questionnaire dropdown must contain **only active and relevant questionnaires**.

If there is only one relevant questionnaire, the UI may show it directly without a dropdown.

---

# 3. Inclusion rules

A questionnaire may appear in the main voter selector only if **all required conditions** are met.

## Required conditions

### 3.1 Valid lifecycle state

The questionnaire state must be one of:

* `published`
* `open`

Do **not** include:

* `draft`
* `saved`
* `closed`
* `ended`
* `counted`
* `archived`

---

### 3.2 Not expired

The questionnaire must not be expired.

Include only if:

* `closed_at` is missing
* or `closed_at` is in the future
* or there is no close timestamp and the questionnaire is still active by state

Exclude if:

* `closed_at` is in the past
* or questionnaire is explicitly marked closed/ended/counted

---

### 3.3 Matches current coordinator context

The questionnaire must belong to the currently configured coordinator context.

Include only if:

* discovered from the currently selected/configured coordinator
* or explicitly restored by ID in the current voter session

Exclude if:

* discovered from another coordinator
* discovered from old unrelated sessions
* discovered globally with no current coordinator relevance

---

### 3.4 Recent enough

The questionnaire must be recent enough to be considered relevant.

Default inclusion window:

* discovered or refreshed within the last **7 days**

Exclude by default if:

* older than 7 days
* unless explicitly restored by ID
* unless explicitly pinned/current

This retention window may be configurable internally, but the default should be short.

---

### 3.5 Not marked stale

Exclude if questionnaire is marked:

* stale
* superseded
* archived
* test-only
* deleted
* invalid

---

# 4. Exclusion rules

The following must **never** appear in the main selector by default:

* questionnaires in `draft`
* questionnaires in `saved`
* questionnaires in `closed`
* questionnaires in `counted`
* questionnaires from unrelated coordinators
* questionnaires from old test runs
* questionnaires loaded only from broad historical local cache
* questionnaires older than the retention window unless manually restored
* duplicate entries for the same questionnaire id

---

# 5. Deduplication rules

Before rendering the selector:

1. dedupe by `questionnaire_id`
2. if multiple records exist for same id:

   * keep the newest valid record
   * prefer current coordinator context
   * prefer active state over inactive state
3. never show the same questionnaire twice

---

# 6. Sorting rules

If multiple questionnaires remain after filtering, sort them in this order:

1. `open` questionnaires first
2. then `published`
3. newest `opened_at` first
4. newest `created_at` or `discovered_at` first
5. finally stable lexical sort by questionnaire id

---

# 7. UI states

## State A - exactly one relevant questionnaire

Show:

* no large dropdown required
* show one selected questionnaire card or compact selector row

Contents:

* title
* description
* questionnaire id
* state

Optional:

* still allow dropdown affordance if implementation simplicity requires it

---

## State B - multiple relevant questionnaires

Show:

* a dropdown containing only filtered relevant questionnaires

Each item should show:

* questionnaire title if available
* questionnaire id
* short state label

Do not show raw giant historical id list.

---

## State C - no relevant questionnaires

Show:

* `No active questionnaire yet.`

Secondary controls:

* `Refresh questionnaire`
* `Restore questionnaire ID`

Do not show an empty historical dropdown full of stale ids.

---

## State D - manually restored questionnaire

If user enters/restores a questionnaire id manually:

* allow it to appear in the selector temporarily
* mark it as restored/current
* do not repopulate the selector with all old history

Optional label:

* `Restored questionnaire`

---

# 8. Restore flow rules

## Restore questionnaire ID input

This is the correct place for historical/manual recovery.

Rules:

* user enters questionnaire id manually
* app attempts to restore/fetch it
* if found and valid, it becomes the current questionnaire
* it may be added to the visible selector for this session
* it must not cause all old questionnaire ids to reappear

## Restore failure

If restore fails:

* show clear error
* do not modify the current selector list except for the current active entries

---

# 9. Local state rules

## Required namespacing

All locally stored questionnaire references must be namespaced by:

* app
* mode
* actor id
* coordinator id where relevant

Recommended shape:

```text id="34895"
app:auditable-voting:voter:{actorId}:{coordinatorId}:{key}
```

Examples:

* current questionnaire id
* recent restored questionnaire ids
* coordinator-linked questionnaire cache

## Forbidden behaviour

Do not build the main selector from:

* all IndexedDB questionnaire records
* all global localStorage questionnaire ids
* all questionnaire ids ever seen by the browser

---

# 10. Data-source rules

## Main selector source

The main selector must be derived from:

* current coordinator-scoped active questionnaire list
* plus current session restored questionnaire if applicable

## Not allowed

The main selector must not be derived from:

* broad historical scans alone
* unfiltered local cache
* mixed-role cache
* unrelated actor state

---

# 11. Rendering rules

Each visible selector item must show:

* questionnaire name if available
* questionnaire id
* state badge (`Published` or `Open`)

Optional:

* opened time
* closing time

Do not show:

* raw ids only, unless name is unavailable
* dozens of rows in a long scroll list
* old irrelevant entries

---

# 12. Acceptance tests

## Test 1 - only active questionnaire visible

Given:

* one `open` questionnaire for current coordinator
* several old `closed` and `draft` questionnaires in local cache

Expect:

* only the open questionnaire appears

---

## Test 2 - unrelated coordinator hidden

Given:

* current coordinator A
* active questionnaire from coordinator B in cache

Expect:

* questionnaire from coordinator B does not appear

---

## Test 3 - old stale questionnaires hidden

Given:

* active-looking but stale questionnaires older than retention window

Expect:

* they do not appear by default

---

## Test 4 - restored questionnaire appears without restoring all history

Given:

* user manually restores questionnaire `q_xyz`

Expect:

* `q_xyz` appears as current/restored
* old unrelated questionnaire ids do not flood the selector

---

## Test 5 - duplicate questionnaire ids deduped

Given:

* multiple cached records for same questionnaire id

Expect:

* only one selector entry shown
* newest valid/current one chosen

---

## Test 6 - no active questionnaire state

Given:

* no relevant questionnaires after filtering

Expect:

* show `No active questionnaire yet`
* show restore control
* do not show giant stale dropdown

---

# 13. Non-negotiable UI outcome

The voter must never see:

* a long scrolling list of historical questionnaire ids from prior tests or sessions
* unrelated coordinator questionnaires
* inactive questionnaires by default

The voter should normally see:

* one current questionnaire
* or a very small list of current questionnaires
* plus a separate restore path for manual recovery

---

# 14. Short implementation summary

Implement the selector as:

1. gather candidate questionnaires from current coordinator-scoped state
2. add current session restored questionnaire if any
3. dedupe by questionnaire id
4. filter by:

   * state in `published|open`
   * not expired
   * current coordinator
   * recent enough
   * not stale/archived/deleted
5. sort by active relevance
6. render:

   * one card if single result
   * short dropdown if multiple
   * empty state if none
