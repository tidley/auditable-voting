Below is a concrete implementation-ready spec for the **mailbox/object-delivery redesign**. It keeps:

* **MLS** for coordinator control
* **public Nostr** for audit
* **private encrypted mailbox objects** for voter request, ticket delivery, and acknowledgement

This is designed specifically to replace the weak **DM ticket/ack pipeline** that is still failing under load, while respecting the fact that the current coordinator OpenMLS path is already live and the main instability is split between **round-1 formation** and **downstream ticket/ack visibility**.

---

# 1. Design goals

This protocol must be:

* **secure**
* **private**
* **decentralised**
* **replayable**
* **browser-first**
* **no backend required for correctness**
* **robust under relay disorder**
* **stable for 1,000+ voters**

It must **not** rely on:

* pairwise DM visibility semantics for ticket delivery
* live-subscription luck for acknowledgement observation
* mutable request/ticket/ack identity
* ad hoc TS-side protocol truth

---

# 2. Protocol planes

## 2.1 Coordinator control plane

Use **MLS** for private coordinator control only.

Carries:

* round proposal
* round approval
* round open readiness
* blind key readiness
* ticket bundle build coordination
* dispute handling
* recovery checkpoints

Implementation:

* Rust-owned
* OpenMLS-backed
* browser coordinator path already exists and must remain the control truth path

## 2.2 Public audit plane

Public Nostr events for:

* election definition
* round definition
* round open
* round close
* public blind key commitments
* receipt commitments
* final result
* public proof bundle

## 2.3 Private mailbox plane

Replace pairwise ticket/ack DMs with:

* request mailbox objects
* ticket mailbox objects
* acknowledgement mailbox objects

This is the new work.

# 3. Invariants

These are mandatory.

## 3.1 Voters are not MLS members

Never put voters into MLS.

## 3.2 One logical request lineage

For one voter in one round:

* one `request_id`
* one `ticket_id`
* one `ack_id`

Retries do **not** create a new logical lineage.

## 3.3 Mailbox plane is object-based, not chat-based

The private plane is not a “conversation”.
It is a set of encrypted objects with stable ids.

## 3.4 Replay is authoritative

All state must be reconstructible by replaying events plus snapshots.

## 3.5 Public audit remains compact

Do not publish internal delivery chatter publicly.
Publish only compact commitments and proof artefacts.

# 4. Identifiers

All ids are lowercase hex unless otherwise stated.

## 4.1 Core ids

### `election_id`

String.
Globally unique per election.

Example:

```json id="j8z3o3"
"election_2026_04_demo_01"
```

### `round_id`

String.
Unique within election.

Example:

```json id="zjjlwm"
"round_001"
```

### `group_id`

String.
For coordinator supervisory group.

Example:

```json id="v63oxr"
"coord_supervisory_main"
```

### `mailbox_id`

Derived per voter per round.

Derivation:

```text id="qqc9y7"
mailbox_id = H("mailbox" || election_id || round_id || voter_pubkey || mailbox_salt)
```

* `mailbox_salt` is round-scoped and private enough to avoid trivial precomputation
* the resulting `mailbox_id` is used as a relay filter tag
* the mailbox contents remain encrypted

## 4.2 Logical lineage ids

### `request_id`

Use the blinded request id directly.

Rule:

```text id="uey8x4"
request_id = blind_request.requestId
```

Do not generate a second top-level logical id.

This is already aligned with the `v0.73` fix and must remain so.

### `ticket_id`

Derived deterministically from request and payload identity.

Rule:

```text id="okjb6g"
ticket_id = H("ticket" || election_id || round_id || request_id || payload_commitment)
```

Where:

* `payload_commitment = H(ticket_payload_canonical_json)`

### `ack_id`

Derived deterministically from `ticket_id`.

Rule:

```text id="4g74ff"
ack_id = H("ack" || election_id || round_id || ticket_id)
```

Retries must not change `ticket_id` or `ack_id`.

# 7. Private mailbox-plane events

These are published to relays but payloads are encrypted.

All mailbox events must include:

* `mailbox_id`
* stable logical ids
* a payload commitment for idempotency
* optional `supersedes_event_id` for retries
* optional `attempt_no`

## 7.1 `mailbox_request_envelope`

### Purpose

Voter submits blinded request privately.

### JSON shape

```json id="47z2kl"
{
  "schema_version": 1,
  "event_type": "mailbox_request_envelope",
  "election_id": "election_2026_04_demo_01",
  "round_id": "round_001",
  "created_at": 1712530250,
  "sender_pubkey": "npub1voter...",
  "payload": {
    "mailbox_id": "hex",
    "request_id": "blindreq_abc123",
    "payload_commitment": "hex",
    "ciphertext_scheme": "nip44v2",
    "ciphertext_type": "blind_request",
    "ciphertext": "base64_or_bech32",
    "attempt_no": 1,
    "supersedes_event_id": null
  }
}
```

### Rules

* `request_id` must equal `blind_request.requestId`
* retry must keep the same `request_id`
* retry may increase `attempt_no`
* retry may set `supersedes_event_id`
* reducers treat all same-`request_id` envelopes as one lineage

## 7.2 `mailbox_ticket_envelope`

### Purpose

Coordinator-side system publishes one encrypted ticket bundle for the voter.

### JSON shape

```json id="g0r8jo"
{
  "schema_version": 1,
  "event_type": "mailbox_ticket_envelope",
  "election_id": "election_2026_04_demo_01",
  "round_id": "round_001",
  "created_at": 1712530300,
  "sender_pubkey": "npub1coord...",
  "payload": {
    "mailbox_id": "hex",
    "request_id": "blindreq_abc123",
    "ticket_id": "hex_ticket_id",
    "payload_commitment": "hex",
    "ciphertext_scheme": "nip44v2",
    "ciphertext_type": "ticket_bundle",
    "ciphertext": "base64_or_bech32",
    "attempt_no": 1,
    "supersedes_event_id": null,
    "coordinator_set_hash": "hex"
  }
}
```

### Rules

* one logical `ticket_id` per logical `request_id`
* retries keep same `ticket_id`
* changed ciphertext due to repackaging is allowed if it decrypts to equivalent logical payload
* reducer accepts first valid visible `ticket_id` lineage and treats retries as same logical ticket

## 7.3 `mailbox_ack_envelope`

### Purpose

Voter acknowledges ticket retrieval/receipt.

### JSON shape

```json id="y45zht"
{
  "schema_version": 1,
  "event_type": "mailbox_ack_envelope",
  "election_id": "election_2026_04_demo_01",
  "round_id": "round_001",
  "created_at": 1712530315,
  "sender_pubkey": "npub1voter...",
  "payload": {
    "mailbox_id": "hex",
    "request_id": "blindreq_abc123",
    "ticket_id": "hex_ticket_id",
    "ack_id": "hex_ack_id",
    "payload_commitment": "hex",
    "ciphertext_scheme": "nip44v2",
    "ciphertext_type": "ticket_ack",
    "ciphertext": "base64_or_bech32",
    "attempt_no": 1,
    "supersedes_event_id": null
  }
}
```

### Rules

* `ack_id` is deterministic from `ticket_id`
* retries keep same `ack_id`
* ack must reconcile even if found only via backfill
* ack may be accepted from a prior attempt if it matches the stable logical lineage

# 10. Reducer state machines

These must live in Rust.

## 10.1 Coordinator control reducer

### State per round

```json id="rd1s6z"
{
  "round_status": "draft|open_proposed|open|closed|published",
  "coordinator_readiness": {
    "mls_join_complete": true,
    "welcome_ack_sent": true,
    "initial_control_backfill_complete": true,
    "auto_approval_complete": true,
    "round_open_publish_safe": true,
    "blind_key_publish_safe": true,
    "ticket_plane_safe": true
  }
}
```

### Rule

Round 1 cannot open unless all required readiness flags are true.

This aligns with the current readiness diagnostics already added in `v0.74`.

## 10.2 Mailbox request reducer

### State per `(round_id, mailbox_id, request_id)`

```json id="qhzp2i"
{
  "request_seen": true,
  "request_first_event_id": "evt1",
  "request_latest_attempt_no": 2,
  "request_payload_commitment": "hex",
  "request_status": "valid|invalid|superseded"
}
```

### Rules

* first valid request lineage wins for that `request_id`
* duplicate same payload is idempotent
* same `request_id` with incompatible payload commitment is invalid and diagnostic-worthy
* retry with same logical payload is allowed

## 10.3 Mailbox ticket reducer

### State per `(round_id, mailbox_id, request_id, ticket_id)`

```json id="s3r3gt"
{
  "ticket_built": true,
  "ticket_published": true,
  "ticket_publish_first_event_id": "evt2",
  "ticket_latest_attempt_no": 2,
  "ticket_visible_to_voter": false,
  "ticket_status": "pending|visible|confirmed|invalid"
}
```

### Rules

* one logical `ticket_id` per `request_id`
* if retry uses same `ticket_id`, reducer merges as one lineage
* if conflicting `ticket_id` appears for same request and incompatible payload, mark invalid

## 10.4 Mailbox ack reducer

### State per `(round_id, mailbox_id, ticket_id, ack_id)`

```json id="g1v2sk"
{
  "ack_expected": true,
  "ack_sent": false,
  "ack_seen_live": false,
  "ack_seen_backfill": false,
  "ack_status": "waiting|seen_live|seen_backfill|invalid|missing_after_recovery"
}
```

### Rules

* `ack_id` must be derived from `ticket_id`
* live and backfill both reconcile into same logical ack state
* duplicate same ack is idempotent
* old valid attempt ids may map to same stable logical ack target

# 11. Ordering rules

One canonical order only.

Sort by:

1. event class precedence
2. `created_at`
3. event id lexical

For mailbox plane, precedence is:

1. `mailbox_request_envelope`
2. `mailbox_ticket_envelope`
3. `mailbox_ack_envelope`

Public and coordinator-plane precedence continues to follow the existing Rust ordering model.

# 12. Recovery rules

## 12.1 Live observation is not enough

If a required state is missing after threshold:

* run bounded backfill
* replay the same reducer
* update state accordingly

## 12.2 Backfill thresholds

### Ticket visibility recovery

If `ticket_published = true` but `ticket_visible_to_voter = false` after threshold:

* fetch mailbox ticket history for `(election_id, round_id, mailbox_id)`
* replay
* if still not visible, resend same `ticket_id`

### Ack recovery

If `ack_expected = true` and `ack_seen_live = false` after threshold:

* fetch mailbox ack history for `(election_id, round_id, mailbox_id, ticket_id)`
* replay
* if valid ack found, mark `ack_seen_backfill = true`
* else mark `missing_after_recovery`

This directly addresses the current ack-side dominant failure mode under load.

# 13. Resend rules

## 13.1 Request resend

Allowed if request not observed or not processed.
Must keep:

* same `request_id`

May change:

* event id
* attempt number

## 13.2 Ticket resend

Allowed if voter did not observe after threshold.
Must keep:

* same `request_id`
* same `ticket_id`

May change:

* event id
* attempt number
* ciphertext encoding if logically equivalent

## 13.3 Ack resend

Allowed if coordinator did not observe after threshold.
Must keep:

* same `ticket_id`
* same `ack_id`

May change:

* event id
* attempt number

# 14. Security and privacy constraints

## 14.1 No security downgrade

This redesign must not weaken:

* confidentiality of private delivery
* replay safety
* coordinator control confidentiality
* audit integrity

## 14.2 Privacy requirements

* mailbox payloads encrypted
* mailbox ids round-scoped and salted
* voters not in MLS
* no public plaintext ticket material
* public audit events compact and commitment-based

## 14.3 Decentralisation requirements

* mailbox objects published to multiple relays
* retrieval via relay replay/backfill
* no server required for correctness
* publisher identity not sole source of truth

# 18. Success criteria

This redesign is working when:

* no pairwise DM semantics are required for ticket delivery
* startup failures are isolated to coordinator readiness only
* ticket delivery works as mailbox retrieval
* ack recovery works via live + backfill
* same logical request/ticket/ack lineage survives resend
* public receipt commitments are deterministic
* Rust remains source of truth

# 19. One-sentence summary

Replace the fragile pairwise DM ticket/ack path with **encrypted round-scoped mailbox objects on relays**, keep **MLS for coordinators**, keep **public Nostr for audit**, and make **Rust reducers authoritative for request, ticket, and ack lineage** so replay, resend, and backfill are deterministic.

# A5. Reducer truth tables

These are the exact reducer rules.

All reducers must be pure and deterministic.

---

## A5.1 Mailbox request reducer truth table

### Key

Reducer key:

* `(election_id, round_id, mailbox_id, request_id)`

### Input event

`mailbox_request_envelope`

### Table

| Current state    | Incoming event condition                                                 | Action                                               | New state                                           |
| ---------------- | ------------------------------------------------------------------------ | ---------------------------------------------------- | --------------------------------------------------- |
| none             | first valid envelope                                                     | create request state                                 | `request_seen=true`, `status=Valid`                 |
| none             | invalid envelope shape                                                   | create diagnostic only                               | state absent or invalid depending on implementation |
| existing valid   | same `request_id`, same `payload_commitment`, higher `attempt_no`        | merge as retry                                       | keep `status=Valid`, update latest attempt          |
| existing valid   | same `request_id`, same `payload_commitment`, duplicate or lower attempt | idempotent ignore                                    | unchanged                                           |
| existing valid   | same `request_id`, different `payload_commitment`                        | reject as conflicting lineage                        | `status=Invalid`, add `ConflictingRequestPayload`   |
| existing invalid | any same-lineage event                                                   | remain invalid unless explicit migration rule exists | unchanged invalid                                   |
| existing valid   | superseding retry with same logical commitment                           | accept and update latest attempt                     | unchanged valid                                     |
| round closed     | any new request                                                          | reject for round closure                             | invalid or ignored with diagnostic                  |

### Notes

* `request_id` is canonical from blind request id
* reducer never creates a second logical request state for same `request_id`

---

## A5.2 Mailbox ticket reducer truth table

### Key

Reducer key:

* `(election_id, round_id, mailbox_id, request_id, ticket_id)`

### Input event

`mailbox_ticket_envelope`

### Table

| Current state                         | Incoming event condition                                       | Action                                      | New state                                                      |
| ------------------------------------- | -------------------------------------------------------------- | ------------------------------------------- | -------------------------------------------------------------- |
| no request state                      | ticket arrives before request known                            | store as pending orphan or reject by policy | implementation-defined, but deterministic                      |
| valid request exists, no ticket state | first valid ticket for request                                 | create ticket state                         | `ticket_built=true`, `ticket_published=true`, `status=Pending` |
| valid ticket exists                   | same `ticket_id`, same commitment, higher `attempt_no`         | merge as retry                              | update latest attempt                                          |
| valid ticket exists                   | same `ticket_id`, same commitment, duplicate                   | idempotent ignore                           | unchanged                                                      |
| valid ticket exists                   | same `request_id`, different `ticket_id`, incompatible payload | reject conflicting lineage                  | mark invalid, add diagnostic                                   |
| ticket published                      | voter observation signal later arrives                         | mark visible                                | `ticket_visible_to_voter=true`, `status=Visible`               |
| ticket visible                        | valid ack later arrives                                        | mark confirmed                              | `status=Confirmed`                                             |
| round closed before ticket valid      | reject or ignore with diagnostic                               | unchanged or invalid                        |                                                                |
| request invalid                       | ticket for invalid request                                     | reject                                      | invalid                                                        |

### Notes

* one logical `ticket_id` lineage per `request_id`
* retries do not create new logical ticket states

---

## A5.3 Mailbox ack reducer truth table

### Key

Reducer key:

* `(election_id, round_id, mailbox_id, ticket_id, ack_id)`

### Input event

`mailbox_ack_envelope`

### Table

| Current state                                                      | Incoming event condition                                           | Action                                   | New state                                                |
| ------------------------------------------------------------------ | ------------------------------------------------------------------ | ---------------------------------------- | -------------------------------------------------------- |
| no ticket state                                                    | ack arrives before ticket known                                    | store pending orphan or reject by policy | deterministic implementation required                    |
| valid ticket exists, no ack state                                  | first valid ack                                                    | create ack state                         | `ack_sent=true`, `ack_seen_live=true`, `status=SeenLive` |
| valid ack exists                                                   | same `ack_id`, same commitment, duplicate                          | idempotent ignore                        | unchanged                                                |
| valid ack exists                                                   | same `ack_id`, higher `attempt_no`                                 | merge as retry                           | unchanged or update attempt                              |
| valid ticket exists                                                | ack found only through backfill                                    | accept as same logical ack               | `ack_seen_backfill=true`, `status=SeenBackfill`          |
| valid ticket exists                                                | same `ticket_id`, wrong `ack_id` not derivable from ticket lineage | reject invalid ack                       | `status=Invalid`, add `InvalidAckLineage`                |
| waiting ack, recovery threshold exceeded, no ack found in backfill | mark missing                                                       | `status=MissingAfterRecovery`            |                                                          |
| confirmed ticket exists                                            | duplicate valid ack later                                          | idempotent ignore                        | unchanged                                                |

### Notes

* ack reducer must accept both live and backfill visibility paths
* acks must reconcile to stable ticket lineage, not latest resend attempt only

---

## A5.4 Ticket observation truth table

This is not a separate Nostr event type unless you choose to add one internally. It is a reducer-side observation state.

### Observation sources

* local voter decryption of `mailbox_ticket_envelope`
* bounded mailbox backfill
* explicit local app state transition after successful decrypt

### Table

| Current state                                            | Observation condition                   | Action                   | New state                                        |
| -------------------------------------------------------- | --------------------------------------- | ------------------------ | ------------------------------------------------ |
| `ticket_published=true`, `ticket_visible_to_voter=false` | voter decrypts ticket live              | mark visible             | `ticket_visible_to_voter=true`, `status=Visible` |
| `ticket_published=true`, `ticket_visible_to_voter=false` | ticket found via backfill and decrypted | mark visible             | `ticket_visible_to_voter=true`, `status=Visible` |
| `ticket_visible_to_voter=true`                           | duplicate observation                   | idempotent ignore        | unchanged                                        |
| `ticket_published=true`                                  | observation timeout exceeded            | trigger bounded backfill | unchanged until backfill result                  |

# A6. Reducer ordering rules for mailbox events

For mailbox-plane replay, use this precedence:

1. `mailbox_request_envelope`
2. `mailbox_ticket_envelope`
3. `mailbox_ack_envelope`

Then within same class:

1. `created_at`
2. event id lexical

This prevents acks from being applied before requests/tickets where timestamps are close but disorder exists.

# A8. Backfill recovery truth rules

## A8.1 Request backfill

If request is missing but later objects exist:

* bounded backfill for that mailbox and round
* replay request events first
* re-apply ticket/ack events after request replay

## A8.2 Ticket backfill

If `publish_succeeded` but `voter_observed=false` after threshold:

* backfill mailbox ticket events
* if matching `ticket_id` found and decryptable, mark `Visible`
* else keep pending and allow bounded resend

## A8.3 Ack backfill

If `ack_expected=true` and `ack_seen_live=false` after threshold:

* backfill mailbox ack events
* if matching `ack_id` found, mark `SeenBackfill`
* else `MissingAfterRecovery`

This is the explicit answer to the current ack-side dominant visibility problem under load.

2. implement mailbox request envelope
3. implement mailbox ticket envelope
4. implement mailbox ack envelope
5. implement Rust mailbox reducers
6. replace existing shard request/ticket/ack DM path
