Below is a formal protocol spec for the **questionnaire-first blind-token architecture**, written in the same style as the earlier protocol work.

It assumes the core design is:

* **public questionnaire definition**
* **private eligibility request**
* **blind-signed one-use response token**
* **ephemeral response key**
* **one response bundle per voter**
* **first valid response per token nullifier wins**
* **response payload may be public or encrypted**

---

# Questionnaire Blind-Token Protocol Specification

Version: `1.0-draft`

## 1. Scope

This protocol defines a Nostr-native questionnaire system where:

* a coordinator publishes a questionnaire publicly
* eligible voters obtain one blind-signed response credential
* each voter submits exactly one response bundle using an ephemeral key
* admission is based on a **token nullifier**
* result summaries are publicly auditable
* payload mode may be:

  * `public`
  * `encrypted`

This protocol is designed for:

* surveys
* course feedback
* governance questionnaires
* multi-question structured polls

It is **not** a per-question voting protocol.
The unit of submission is **one response bundle per questionnaire**.

---

## 2. Terminology

### Questionnaire

A public object containing metadata and a list of questions.

### Response bundle

A single submission containing answers for all or some questions in a questionnaire.

### Blind token

A coordinator-issued blind signature over a one-use response token, unlinked from the eventual submission identity.

### Token nullifier

A stable one-use identifier derived from the response token. The verifier uses it to reject duplicate submissions.

### Ephemeral response key

A fresh Nostr keypair used only to sign one response submission.

### Payload mode

Defines whether answers are:

* `public`
* `encrypted`

---

## 3. Protocol objectives

The protocol MUST provide:

* one public questionnaire definition
* one blind-signed response token per eligible voter
* one response bundle per token nullifier
* deterministic duplicate rejection
* deterministic verifier rules
* public result summary publication

The protocol SHOULD provide:

* browser-native operation
* client-side verifier support
* relay-compatible public distribution of questionnaire and summary objects

The protocol MAY provide:

* encrypted response payloads
* coordinator-only decryption
* public response payload mode for fully public verification

---

## 4. Event kinds

Suggested custom event kinds:

* `34500` - questionnaire definition
* `34501` - questionnaire state
* `34502` - blind-token response submission
* `34503` - questionnaire result summary

Private eligibility/issuance transport is not required to use public feed event kinds and may use a private transport layer.

---

## 5. Questionnaire definition event

## 5.1 Purpose

Defines a questionnaire and its validation rules.

## 5.2 Kind

`34500`

## 5.3 Event type

`questionnaire_definition`

## 5.4 Content schema

```json
{
  "schema_version": 1,
  "event_type": "questionnaire_definition",
  "questionnaire_id": "course_feedback_2026_term1",
  "title": "Course feedback",
  "description": "Please answer all required questions.",
  "author_pubkey": "npub1coordinator...",
  "created_at": 1712530000,
  "open_at": 1712533600,
  "close_at": 1712619999,
  "response_mode": "blind_token",
  "response_payload_mode": "encrypted",
  "blind_signing_pubkey": "npub1blindsign...",
  "coordinator_encryption_pubkey": "npub1encrypt...",
  "verifier": {
    "type": "questionnaire_blind_v1",
    "version": "1.0.0",
    "code_hash": "sha256:abc123..."
  },
  "questions": [
    {
      "question_id": "q1",
      "type": "yes_no",
      "prompt": "Was the course material clear?",
      "required": true
    },
    {
      "question_id": "q2",
      "type": "multiple_choice",
      "prompt": "How would you rate the pace?",
      "required": true,
      "multi_select": false,
      "options": [
        { "option_id": "slow", "label": "Too slow" },
        { "option_id": "good", "label": "About right" },
        { "option_id": "fast", "label": "Too fast" }
      ]
    },
    {
      "question_id": "q3",
      "type": "free_text",
      "prompt": "What should be improved?",
      "required": false,
      "max_length": 1000
    }
  ]
}
```

## 5.5 Required tags

```json
[
  ["t", "questionnaire"],
  ["questionnaire", "course_feedback_2026_term1"],
  ["schema", "1"],
  ["etype", "questionnaire_definition"],
  ["response_mode", "blind_token"],
  ["payload_mode", "encrypted"]
]
```

## 5.6 Validation rules

A questionnaire definition is valid iff:

* `questionnaire_id` is present
* `schema_version` is supported
* `title` is non-empty
* `open_at < close_at`
* all `question_id`s are unique
* all multiple-choice `option_id`s are unique within their question
* `blind_signing_pubkey` is present
* `coordinator_encryption_pubkey` is present when `response_payload_mode = "encrypted"`

## 6.1 Purpose

Marks state transitions such as opening and closing.

## 6.2 Kind

`34501`

## 6.3 Event type

`questionnaire_state`

## 6.4 Content schema

```json
{
  "schema_version": 1,
  "event_type": "questionnaire_state",
  "questionnaire_id": "course_feedback_2026_term1",
  "state": "open",
  "created_at": 1712533600,
  "author_pubkey": "npub1coordinator..."
}
```

Allowed states:

* `draft`
* `open`
* `closed`
* `results_published`

## 6.5 Required tags

```json
[
  ["t", "questionnaire_state"],
  ["questionnaire", "course_feedback_2026_term1"],
  ["schema", "1"],
  ["etype", "questionnaire_state"],
  ["state", "open"]
]
```

## 6.6 State rule

If state events are used, the latest valid state event by the coordinator for that questionnaire wins.

## 7. Eligibility request object

## 7.1 Purpose

Allows a voter to request one response credential for a questionnaire.

## 7.2 Transport

Private transport.
This object is not required to be a public feed event.

## 7.3 Object schema

```json
{
  "schema_version": 1,
  "event_type": "questionnaire_eligibility_request",
  "questionnaire_id": "course_feedback_2026_term1",
  "request_id": "req_001",
  "created_at": 1712535000,
  "requester_pubkey": "npub1longtermvoter...",
  "blinded_token_message": "base64..."
}
```

## 7.4 Validation rules

A request is valid iff:

* `questionnaire_id` references an existing questionnaire
* questionnaire is not closed
* `blinded_token_message` is present
* `request_id` is present

The eligibility policy itself is out of scope for the public verifier.

---

## 8. Blind issuance object

## 8.1 Purpose

Returns a blind signature for a one-use response token.

## 8.2 Transport

Private transport.

## 8.3 Object schema

```json
{
  "schema_version": 1,
  "event_type": "questionnaire_blind_issue",
  "questionnaire_id": "course_feedback_2026_term1",
  "request_id": "req_001",
  "created_at": 1712535200,
  "issuer_pubkey": "npub1blindsign...",
  "blinded_signature": "base64..."
}
```

## 8.4 Validation rules

A blind issue object is valid iff:

* `questionnaire_id` matches the request
* `request_id` matches a request
* `blinded_signature` is present

Unblinding occurs client-side.

## 9. Blind response submission event

## 9.1 Purpose

Submits exactly one questionnaire response bundle under an anonymous one-use credential.

## 9.2 Kind

`34502`

## 9.3 Event type

`questionnaire_response_blind`

## 9.4 Common content schema

```json
{
  "schema_version": 1,
  "event_type": "questionnaire_response_blind",
  "questionnaire_id": "course_feedback_2026_term1",
  "response_id": "resp_001",
  "submitted_at": 1712537200,
  "author_pubkey": "npub1_ephemeral...",
  "token_nullifier": "nullifier_xyz",
  "token_proof": {
    "token_commitment": "commitment_abc",
    "questionnaire_id": "course_feedback_2026_term1",
    "signature": "unblinded_signature"
  }
}
```

## 9.5 Public payload mode

When `response_payload_mode = "public"`, the event MUST include:

```json
{
  "answers": [
    {
      "question_id": "q1",
      "answer_type": "yes_no",
      "value": true
    },
    {
      "question_id": "q2",
      "answer_type": "multiple_choice",
      "selected_option_ids": ["good"]
    },
    {
      "question_id": "q3",
      "answer_type": "free_text",
      "text": "More worked examples would help."
    }
  ]
}
```

## 9.6 Encrypted payload mode

When `response_payload_mode = "encrypted"`, the event MUST include:

```json
{
  "encrypted_payload": "base64...",
  "payload_hash": "sha256:def456..."
}
```

and MUST NOT include plaintext `answers`.

## 9.7 Required tags

```json
[
  ["t", "questionnaire_response_blind"],
  ["questionnaire", "course_feedback_2026_term1"],
  ["schema", "1"],
  ["etype", "questionnaire_response_blind"],
  ["nullifier", "nullifier_xyz"],
  ["e", "<questionnaire_definition_event_id>"]
]
```

## 9.8 Signature rule

The event MUST be signed by the ephemeral response key.

The ephemeral key MUST be distinct from the long-term requester identity key unless the implementation explicitly chooses otherwise.

## 12. Verifier rules

The verifier operates over:

* one questionnaire definition
* zero or more questionnaire state events
* zero or more blind response events
* optionally a result summary event for comparison

## 12.1 Effective questionnaire state

A questionnaire is open iff:

* current time is within `open_at` and `close_at`
* and no later valid state event marks it closed

Implementations MAY use the explicit state event if present.

## 12.2 Response admission

A blind response is accepted iff:

1. questionnaire definition exists
2. questionnaire is open
3. response event shape is valid
4. `token_proof` is valid against `blind_signing_pubkey`
5. `token_proof.questionnaire_id` matches the questionnaire
6. `token_nullifier` has not already been accepted
7. payload mode rules are satisfied
8. answer validation succeeds

## 12.3 Duplicate rule

Responses MUST be ordered by:

1. `event.created_at` ascending
2. `event.id` ascending

Then:

> **first valid response per `tokenNullifier` wins**
> later valid responses with the same `tokenNullifier` MUST be rejected as `duplicate_nullifier`

## 12.4 Public payload answer validation

When `response_payload_mode = "public"`, a response is answer-valid iff:

* all required questions are answered
* every `question_id` exists
* `answer_type` matches the question definition
* every selected option id exists
* free-text length is within `max_length`

## 12.5 Encrypted payload validation

When `response_payload_mode = "encrypted"`:

### Public verifier MAY validate only:

* token proof validity
* nullifier uniqueness
* outer event shape

### Coordinator-side verifier MUST additionally validate:

* decryption succeeded
* decrypted answer payload matches schema
* required answers present
* answer types valid
* options valid
* text lengths valid

## 12.6 Rejection reasons

Recommended rejection reasons:

* `questionnaire_not_found`
* `questionnaire_not_open`
* `questionnaire_closed`
* `invalid_token_proof`
* `duplicate_nullifier`
* `invalid_response_shape`
* `unknown_question_id`
* `missing_required_answer`
* `wrong_answer_type`
* `invalid_option_id`
* `free_text_too_long`
* `decryption_failed`

---

## 13. Admission evaluator rule

The canonical admission evaluator SHOULD implement:

```text
sort responses by (created_at asc, event_id asc)
for each response:
  if token proof invalid -> reject
  else if nullifier already accepted -> reject duplicate_nullifier
  else if payload invalid -> reject
  else accept
```

This is the decisive rule for adversarial duplicate submissions.

## 14. UI display model

The UI should make the system feel like a native questionnaire post while still exposing verifier truth.

## 14.1 Feed card view

A questionnaire definition SHOULD render like a structured post.

Display:

* title
* description
* number of questions
* open / closed state
* response mode badge:

  * `Blind token`
* payload mode badge:

  * `Encrypted` or `Public`
* verifier badge:

  * `Verifiable`
* accepted response count if available

Suggested compact card:

* title
* short description
* open/close status
* `25 responses`
* `View questionnaire`

## 14.2 Expanded questionnaire view

Display:

* title
* instructions
* question list
* progress indicator
* payload/privacy note
* submit button

For voters:

* show questionnaire content first
* then token acquisition state
* then response form
* then submission state

## 14.3 Token acquisition UI

For blind-token mode, show a small stepper:

1. `Request token`
2. `Token issued`
3. `Response key ready`
4. `Submit response`

Status messages:

* `Waiting for coordinator issuance`
* `Token ready`
* `Response submitted`

Do not expose raw cryptographic details by default.
Advanced debug may show:

* token nullifier prefix
* ephemeral pubkey prefix

## 14.4 Response form UI

Question rendering rules:

### Yes/no

* segmented control or radio buttons

### Multiple choice

* radio buttons for single-select
* checkboxes for multi-select

### Free text

* textarea
* character counter if `max_length` present

All questions are filled into one bundle.
There should be only one final submit action.

## 14.5 Coordinator console

The coordinator view SHOULD show:

* questionnaire published / open status
* accepted response count
* rejected response count
* duplicate-nullifier count
* result summary publication status
* optional decrypted export availability in encrypted mode

Suggested summary panel:

* `Questionnaire published`
* `25 accepted`
* `0 rejected`
* `0 duplicate nullifiers`
* `Result summary published`

## 14.6 Verifier / audit UI

Clients SHOULD be able to show:

* `Published summary matches local verifier`
* or `Summary mismatch`
* `Accepted responses: N`
* `Rejected responses: M`
* duplicate-nullifier count
* validation issues if any

For encrypted mode, clients should clearly distinguish:

* `Publicly verifiable admission`
* `Coordinator-only answer decryption`

## 14.7 Result display

In public payload mode:

* show full aggregate charts
* optionally allow browsing public responses

In encrypted payload mode:

* show only aggregate summaries publicly
* free-text answers should not be displayed publicly by default

## 15. Recommended user flow

### Coordinator

1. Create questionnaire
2. Publish questionnaire definition
3. Open questionnaire
4. Wait for responses
5. Publish result summary

### Voter

1. Open questionnaire
2. Read questions
3. Request response token
4. Receive token
5. Fill answers
6. Submit one response bundle

---

## 16. Implementation guidance

For relay compatibility:

* use broad fetch by kind when critical
* reconcile locally by `questionnaire_id` and `event_type`
* do not rely on relay tag-index behaviour alone for correctness

For correctness:

* treat the admission evaluator as the canonical source of duplicate-nullifier truth
* keep event ordering deterministic
* keep one questionnaire = one response bundle

---

## 17. Security and privacy notes

This protocol provides:

* unlinkability between long-term eligibility request identity and final response identity, assuming blind-signature issuance is sound
* one-use response admission via `tokenNullifier`
* optional private answers via encrypted payload mode

It does not by itself guarantee:

* hidden participation from all network observers
* threshold trust distribution
* public verifiability of encrypted answer contents

Those are later extensions.

---

## 18. Summary of normative rules

1. A questionnaire definition MUST be public.
2. A voter MUST use exactly one blind-signed token per response submission.
3. A response submission MUST carry one `tokenNullifier`.
4. The verifier MUST accept at most one response per `tokenNullifier`.
5. Responses MUST be ordered canonically by `(created_at, event_id)`.
6. The earliest valid response for a nullifier MUST be accepted.
7. Later responses for the same nullifier MUST be rejected as `duplicate_nullifier`.
8. One response submission MUST contain the full response bundle for the questionnaire.
9. Result summaries MUST be derived from accepted submissions only.
