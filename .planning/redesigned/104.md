The coordinator config screen is trying to do setup, drafting, publishing, coordination, and monitoring all at once. It feels chaotic.

## Core problems

1. **Wrong first-load state**

   * You land in a fully-expanded admin console.
   * It shows draft questions, coordinator controls, voter/coordinator areas, and stats before the user has done anything.
   * Default questions are especially bad because they look real.

2. **No clear workflow**

   * The intended sequence should be:

     1. create questionnaire
     2. add questions
     3. invite / collect voters
     4. publish
     5. monitor responses
     6. close
   * Right now, publish/open/close/results/buttons all coexist too early.

3. **Poor hierarchy**

   * Everything has similar weight.
   * Questions do not read as distinct cards.
   * Secondary controls look as important as primary actions.

4. **Action ambiguity**

   * Buttons appear active without clear preconditions.
   * Some buttons seem to do nothing, or their result is not visible.
   * “Publish”, “Open”, “Refresh”, “Publish results” are too close together and easy to misunderstand.

5. **Question editing is cramped**

   * Question blocks visually merge together.
   * Reorder / duplicate / delete are noisy and repeated too often.
   * The user has to parse too much chrome around each question.

---

# Recommended redesign

## 1. Replace the current first-load screen

On first load, show a simple **draft setup page** only.

### First-load layout

**Header**

* Questionnaire title
* Status badge: `Draft`

**Main panel**

* Questionnaire name
* Short description
* Close-after time
* Empty question list

**Primary CTA**

* `Add first question`

**Secondary CTA**

* `Invite voters`

### Hide completely on first load

* stats
* results
* coordinator management
* sub-coordinators
* voter list
* publish results
* open / close controls
* QR

These should appear only when relevant.

---

## 2. Remove all default questions

There should be **zero** questions by default.

Empty state text:

> No questions yet. Add your first question to begin building the questionnaire.

Buttons:

* `Add text question`
* `Add multiple choice question`
* `Add free text question`

That is it.

---

## 3. Make the questionnaire builder card-based

Each question should be a distinct large card with strong separation.

### Question card structure

**Top row**

* `Question 1`
* type badge: `Text` / `Multiple choice` / `Free text`
* drag handle
* overflow menu `...`

**Body**

* Prompt
* Required toggle
* Type-specific fields

**Bottom row**

* `Duplicate`
* `Move up`
* `Move down`
* `Delete`

Do **not** scatter buttons around the card.

### Visual changes

* much more vertical spacing between question cards
* darker card background or border contrast
* 20 to 24 px padding inside each card
* 24 to 32 px margin between cards

---

## 4. Replace many visible buttons with staged actions

The main mistake is exposing actions before they make sense.

## Draft stage

Visible:

* `Save draft`
* `Invite voters`
* `Preview questionnaire`

Disabled:

* `Publish questionnaire`

Helper text:

> Publishing is available after at least 1 voter is known and the questionnaire has at least 1 question.

## Ready-to-publish stage

Visible:

* `Publish questionnaire`

Secondary:

* `Preview`
* `Edit details`

## Published but not open

Visible:

* `Open questionnaire`

Secondary:

* `Copy link`
* `Show QR`

## Open stage

Visible:

* `Close questionnaire`

Secondary:

* `View live responses`

## Closed stage

Visible:

* `Publish results`

This makes the workflow obvious.

---

## 5. Make preconditions explicit

At the moment it sounds like questions may be getting published too early.

That should be impossible.

## Publishing rules

`Publish questionnaire` should only enable if:

* questionnaire title is filled
* at least 1 question exists
* at least 1 voter is known
* all required question fields are valid

If not enabled, show a checklist:

* `Questionnaire title added`
* `At least one question added`
* `At least one voter invited`
* `All questions valid`

This removes ambiguity.

---

## 6. Move coordination and voter management out of the main editing flow

These are advanced sections and should not sit in the middle of authoring.

## Use tabs or steps

Top-level tabs should be:

* `Build`
* `Audience`
* `Publish`
* `Responses`
* `Results`

### Build

Questionnaire details and question editing only.

### Audience

* voters
* sub-coordinators
* QR / link if relevant for inviting

### Publish

* readiness checklist
* publish/open/close actions
* current lifecycle status

### Responses

* live accepted/rejected counts
* latest response activity

### Results

* result publication and final outputs

This is much more logical than the current mixed page.

---

## 7. Fix QR behaviour by making it visible and testable

If `Show QR` currently does nothing, that is a high-priority bug.

### Correct behaviour

Clicking `Show QR` should open one of:

* modal dialog
* side panel
* inline expanded area

The QR should encode a clear, single-purpose URL:

* either invite link
* or questionnaire participation link

It should include:

* a visible text link under it
* `Copy link`
* `Download QR`

If QR is unavailable, show:

> QR available after questionnaire is published.

Do not show a dead button.

---

## 8. Reduce the number of always-visible sections

Current sections like:

* private questionnaire
* coordination
* status
* coordinator management
* sub-coordinators
* voters

should not all be open together.

### Better rule

Only one major section expanded at a time.

Or use tabs, which is better.

If you keep accordions:

* default open only the current task section
* everything else collapsed

For example in draft mode:

* `Questionnaire` open
* `Questions` open
* `Voters` collapsed
* `Coordination` collapsed
* `Results` hidden

---

## 9. Improve wording

Some labels are too technical or vague.

### Replace

* `Private questionnaire` -> `Questionnaire draft`
* `Publish questionnaire` -> `Publish draft`
* `Open questionnaire` -> `Start accepting responses`
* `Close questionnaire` -> `Stop accepting responses`
* `Publish results` -> `Publish summary results`
* `Coordination` -> `Coordinator status`
* `Sub-coordinators` -> `Additional coordinators`
* `Voters` -> `Invited voters`

Use labels that describe an action outcome.

---

## 10. Recommended final screen structure

## Draft mode

### Header

* Title
* status badge `Draft`

### Main content

#### Section 1 - Questionnaire details

* title
* description
* close-after minutes

#### Section 2 - Questions

* empty state or question cards
* add-question buttons

#### Section 3 - Readiness

* checklist for publish

### Footer action bar

* `Save draft`
* `Preview`
* `Invite voters`
* `Publish draft` disabled/enabled

---

## Published/open mode

### Header

* title
* status badge `Open`
* primary action `Close questionnaire`

### Main content tabs

* `Responses`
* `Audience`
* `Questionnaire`
* `Results`

Do not keep editing controls mixed into the monitoring screen.

---

# Highest-priority fixes

If you want the shortest route to “much better”, do these first:

1. remove default questions
2. hide all advanced sections on first load
3. split the page into `Build / Audience / Publish / Responses / Results`
4. make question cards much larger and visually separated
5. disable publish/open actions until prerequisites are satisfied
6. fix `Show QR` so it opens a visible modal or panel
7. remove dead or unclear buttons

---

# A good mental model

The current page behaves like an internal protocol dashboard.

The user needs a **wizard-like questionnaire builder** with admin/status features available later.

That means:

* **author first**
* **invite second**
* **publish third**
* **monitor fourth**
* **result fifth**

not all at once.
