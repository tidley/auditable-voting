Here is the concrete decision set I would choose for this repo now, to unblock implementation and avoid reopening the architecture.

# Recommended decisions

## 1. Blind proof verification algorithm

Use **single-signer blind Schnorr over secp256k1**, tied to the questionnaire by an explicit domain separator.

### Rule

* `tokenProof.signature` = unblinded Schnorr signature
* verification key = `blind_signing_pubkey` from questionnaire definition
* signed message = canonical hash of blinded-token message preimage fields
* hash function = SHA-256
* domain separator string:

  * `auditable-voting/questionnaire-blind-token/v1`

### Canonical signed message preimage

```text
domain = "auditable-voting/questionnaire-blind-token/v1"
message = sha256(json_canonical({
  "questionnaire_id": questionnaireId,
  "token_secret_commitment": tokenSecretCommitment,
  "response_mode": "blind_token",
  "schema_version": 1
}))
```

### Why

* secp256k1 fits the Nostr ecosystem
* single-signer keeps MVP simple
* Schnorr keeps later threshold upgrade path open

---

## 2. Nullifier derivation rule

Derive `tokenNullifier` deterministically from the unblinded token secret and questionnaire id.

### Rule

```text
tokenNullifier = sha256(
  json_canonical({
    "domain": "auditable-voting/questionnaire-nullifier/v1",
    "questionnaire_id": questionnaireId,
    "token_secret": tokenSecret
  })
)
```

### Requirements

* `token_secret` is locally generated client-side random 32 bytes
* nullifier must be stable for a given token and questionnaire
* same token cannot be reused across questionnaires without producing a different nullifier

### Why

* simple
* deterministic
* questionnaire-bound
* no dependence on ephemeral response pubkey

---

## 3. Issuance transport contract

Use the **existing private envelope/mailbox transport** for:

* `questionnaire_eligibility_request`
* `questionnaire_blind_issue`

Do **not** invent a second private transport right now.

### Rule

* issuance path stays private
* questionnaire definition / response / result remain on public questionnaire transport
* internal event names may remain current while spec-level naming is introduced above them

### Why

* lowest implementation risk
* current private path already exists
* avoids transport churn while core questionnaire path is stabilising

---

## 4. Mode authority

Run **both paths behind a mode flag**, but make the questionnaire flow authoritative for new questionnaire objects.

### Rule

* questionnaire flow admission mode:

  * `blind_token` = new authoritative mode
  * `legacy_private_envelope` = transitional / compatibility mode only
* new questionnaire definitions must explicitly declare:

  * `response_mode = blind_token`

### Why

* avoids ambiguous runtime branching
* preserves migration room
* keeps new code path clean

---

## 5. Event kind finalisation

**Keep current deployed questionnaire kinds for now** and map them to the formal spec in docs.

### Rule

* implementation keeps current kinds `14120..14124` for compatibility
* spec documents target kinds `34500..34503`
* add a single constants module that aliases both:

  * `IMPLEMENTATION_KIND_*`
  * `SPEC_KIND_*`

### Why

* avoids breaking working relay behaviour
* lets docs/spec stabilise without forcing migration immediately

---

## 6. Coordinator model for questionnaires

Use **single-signer only** for questionnaire blind issuance in this repo version.

### Rule

* one coordinator signs questionnaire tokens
* no threshold blind issuance yet
* no multi-coordinator issuance semantics in MVP

### Why

* current working path is single coordinator
* threshold issuance is a separate project-sized addition
* keep protocol stable first

---

## 7. Encrypted payload verifier boundary

In encrypted mode, make **admission publicly verifiable** and **answer validity coordinator-verifiable**.

### Publicly verifiable

* questionnaire exists
* questionnaire open
* response event shape valid
* token proof valid
* nullifier unique
* payload hash present

### Coordinator-only verifiable

* decrypt success
* answer schema validity
* required question completion
* option validity
* free-text length

### Rule

Do **not** add extra commitments beyond `payloadHash` in MVP.

### Why

* simple and honest boundary
* matches actual trust model
* avoids fake public verifiability claims

---

## 8. Result hash definition

Define `result_hash` as SHA-256 over canonical JSON of the final computed public result summary.

### Canonical input

```text
result_hash = sha256(json_canonical({
  "questionnaire_id": questionnaireId,
  "accepted_response_count": acceptedCount,
  "rejected_response_count": rejectedCount,
  "accepted_nullifier_count": acceptedNullifierCount,
  "question_summaries": canonicalQuestionSummaries
}))
```

### Canonical ordering

* questions sorted by `question_id`
* option counts sorted by `option_id`
* all numeric values encoded as JSON numbers
* no extra whitespace
* UTF-8 encoding

### Why

* easy to reproduce in TS and Rust
* enough for MVP integrity
* compatible with public summary verification

---

## 9. State precedence semantics

Explicit `questionnaire_state` events override inferred time-window state.

### Rule

1. If explicit state exists, latest valid coordinator-authored state wins.
2. If no explicit state exists:

   * open iff `now >= open_at && now < close_at`
3. If explicit state conflicts with timestamps, explicit state wins.

### Why

* gives coordinator operational control
* avoids ambiguity during testing
* easy to explain and implement

---

## 10. Migration / backward compatibility

Treat all existing deployed questionnaire data as **legacy mode objects** unless explicitly marked `blind_token`.

### Rule

* old objects keep old interpretation
* new blind-token logic only applies when:

  * questionnaire definition declares `response_mode = blind_token`
* do not try to reinterpret historical responses as blind-token submissions

### Why

* safest migration boundary
* no silent behaviour change
* avoids corrupting old data semantics

# Summary: the unblocking decision set

If I compress this to the minimum necessary decisions:

* **Crypto**: single-signer blind Schnorr on secp256k1
* **Nullifier**: SHA-256 over questionnaire-bound token secret
* **Issuance transport**: reuse existing private mailbox/envelope path
* **Mode**: blind-token is authoritative for new questionnaires
* **Kinds**: keep current implementation kinds for now
* **Coordinator model**: single signer only
* **Encrypted mode**: public admission verification, coordinator-only payload verification
* **Result hash**: SHA-256 over canonical summary JSON
* **State precedence**: explicit state beats timestamps
* **Migration**: old data remains legacy
