# Likely bug

The likely bug is in:

* `web/src/QuestionnaireCoordinatorPanel.tsx`

The pattern suggests:

1. panel mounts before coordinator identity is available
2. readiness state is computed once
3. later identity update does not fully re-trigger publish readiness
4. with more voters / more load, the timing window widens and the panel stays stuck in “key not loaded”

This is a classic stale-derived-state / missing-effect-dependency problem.

# Most likely concrete fix

## In `QuestionnaireCoordinatorPanel.tsx`

### 1. Treat coordinator identity as asynchronous and authoritative

Do not derive a permanent “not ready” state on first render.

Use a live value like:

```ts
const coordinatorPubkey = ...;
const coordinatorKeyReady = Boolean(coordinatorPubkey);
```

and compute all publish gating from that current value, not from an earlier latched boolean.

### 2. Re-run readiness effects when identity changes

Any effect that gates:

* questionnaire publish
* questionnaire open publish
* result publish

must depend on:

* `coordinatorPubkey`
* `coordinatorKeyReady`
* `questionnaireId`
* any other required runtime inputs

If there is an effect like:

```ts
useEffect(() => {
  if (!coordinatorKeyReady) return;
  ...
}, []);
```

that is the bug.

It should depend on the key readiness inputs.

### 3. Remove stale “loaded once” coordinator key state

If you have state like:

```ts
const [coordinatorKeyLoaded, setCoordinatorKeyLoaded] = useState(false);
```

and it is only set in one startup effect, replace it with derived state or ensure it updates whenever the identity source updates.

### 4. Fail closed, but retry automatically

If key is missing, show not-ready state, but keep polling/re-reading until:

* key present
* panel debug object updated
* publish path can start

Do not require a manual refresh.
