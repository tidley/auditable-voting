That is a very useful result. You now have a clean, evidence-backed bottleneck:

> **The dominant miss stage is `ticket_not_observed`.**

Not:

Your new data says:

# Next phases

## Phase 5 - Fix `ticket_not_observed`

## Objective

Increase the rate at which issued tickets are actually observed by voters.

## Why

This is now the dominant miss stage in both:

* repeated `1 / 5 / 1`
* sampled `1 / 25 / 1`

## Working hypothesis

Ticket events are being:

* published successfully sometimes but not seen promptly by the intended voter
* or published too late / too narrowly to be observed in time
* or not retried/recovered aggressively enough when initial observation fails

## Required work

### 5.1 Add explicit ticket visibility recovery

For each ticket lineage:

* if `ticketSent = true`
* and `ticketObserved = false` after a short threshold

then:

* run bounded mailbox/ticket backfill for that voter/round
* replay the retrieved mailbox events
* if still not visible, resend the same logical `ticket_id`

Important:

* **same `ticket_id`**
* no new logical lineage
* resend must be idempotent

### 5.2 Separate these states clearly

For each row / voter:

* `ticketBuilt`
* `ticketPublishStarted`
* `ticketPublishSucceeded`
* `ticketObservedLive`
* `ticketObservedBackfill`
* `ticketResent`
* `ticketStillMissing`

Right now you mostly know `ticketSent` vs `ticketObserved`. Add the intermediate stages.

### 5.3 Record relay subset used for each ticket send

For every ticket publish attempt, log:

* relays targeted
* relays that confirmed write success
* whether later observation came from live subscription or backfill

This will tell you whether the miss clusters by relay choice.

## Exit condition

Repeated `1 / 5 / 1` runs no longer fail primarily on `ticket_not_observed`.

---

## Phase 6 - Bound and tune ticket send concurrency

## Objective

Stop bursty send behaviour from overwhelming visibility.

## Why

At `1 / 25 / 1`, only **8 tickets were observed** and **7 ballots were accepted**. That is consistent with a delivery plane that is dropping or delaying visibility under burst load.

## Required work

### 6.1 Add a small send concurrency window

Do **not** blast all pending tickets at once.

Use:

* max 2 to 4 concurrent ticket publishes per coordinator in single-coordinator mode

### 6.2 Add per-ticket retry delay with jitter

If not observed after threshold:

* backfill first
* then resend with same `ticket_id`
* stagger retries with jitter

### 6.3 Cap total retry count

For first deployment:

* small bounded retry count, e.g. 2 or 3

### 6.4 Keep one outstanding active send per voter/ticket lineage

Do not allow multiple simultaneous sends for the same logical ticket lineage.

## Exit condition

`1 / 25 / 1` observation rate rises materially above the current 8/25 range.

## Objective

Turn ticket observation into a first-class measured pipeline.

## Required work

Extend the harness and/or coordinator debug to capture:

For each unmatched row:

* `ticketPublishStartedAt`
* `ticketPublishSucceededAt`
* `ticketObservedLiveAt`
* `ticketObservedBackfillAt`
* `ticketResentCount`
* `ticketRelayTargets`

Then summarise:

* backfill-observed count

## Why

You now know the miss stage. The next step is to know whether the fix comes from:

* relay widening
* better backfill
* slower/bounded send rate
* or all three

## Exit condition

Every `ticket_not_observed` case can be classified as:

* recovered by backfill
* recovered by resend
* still missing after retries

## Phase 9 - Tighten the success threshold for the first deployment

## Objective

Make the deployment criterion realistic but still strict enough.

## Why

Right now success is binary and useful, but for initial course feedback you likely want:

* a strong success target
* but not one that hides real improvement

## Suggested threshold

For internal testing:

* track full pass/fail
* also track acceptance ratio

For example:

* hard pass = 25/25 accepted
* soft pass = high acceptance ratio with zero rejections and all misses attributable to ticket visibility

Do **not** lower standards permanently, but do record both.

## Exit condition

You can tell whether a run failed because:

* protocol correctness broke
* or because delivery reliability is not yet high enough

Right now the latter is clearly the case.
