System Architecture

Voters interact with a web portal to obtain blind-signed Cashu proofs, publish votes to Nostr relays, and submit proofs privately to the coordinator for burning and tallying.

Phase 1 — Issuance

Proof Issuance Flow

A voter proves eligibility by signing a claim with their real npub. The coordinator verifies eligibility and auto-approves the mint quote via gRPC. Blind issuance ensures the coordinator cannot link the proof to the voter's identity at vote time.

Phase 2 — Voting

Voting & Proof Submission

The voter generates an ephemeral Nostr keypair, publishes a structured ballot publicly (kind 38000), then submits the proof privately via a NIP-17 gift wrap. The coordinator burns the proof on the mint and publishes an acceptance receipt.

Phase 3 — Tally

Tally Pipeline & Merkle Audit

When the election closes, the coordinator publishes commitment roots and the final signed result. Every voter can verify their vote is included via a Merkle inclusion proof. Supply invariants (issued ≤ max_supply, spent ≤ issued) prevent phantom votes.

Protocol

Security

Trust Model

The system is not trustless, but is auditable with bounded cheating capability. Cryptographic guarantees constrain what the coordinator can do; the remaining trust assumptions are narrow and monitorable.

Cryptographically Enforced

  • 1 proof = 1 vote (spent_set)
  • issued ≤ max_supply (hard cap)
  • spent ≤ issued (commitment roots)
  • spent ⊂ issued (subset proofs)
  • Vote inclusion via Merkle proof
  • Blind issuance breaks identity link
  • Ephemeral keys hide voter identity

Publicly Verifiable

  • All events on Nostr relays
  • Eligibility set (kind 38009)
  • Issuance receipts (kind 38011)
  • Acceptance receipts (kind 38002)
  • Commitment roots (38005, 38006)
  • Final result (kind 38003)
  • HTTP API for convenience only

Trust Assumptions

  • Fair issuance policy
  • Non-censorship of valid voters
  • Honest timing adherence
  • Coordinator key not compromised
  • Relay data availability
  • ≥ 2 of 3 mints honest
Isolation

Per-Election Mint Keysets

Each election uses a unique mint mnemonic derived from the coordinator's nsec and the election ID. Old proofs become mathematically invalid after a keyset rotation because the signing keys don't exist in the new mint database.

Censorship Resistance

3-Mint 2-of-3 Quorum Model

Each voter obtains blind-signed proofs from three independent mints. A vote is valid if proofs from at least 2-of-3 mints are valid and unspent. This raises the attack threshold from 1 to 2 colluding entities.
