Why auditable anonymous voting is hard, and why this stack can make it work

The usual failure mode is simple: either the operator can tell who voted for what, or outsiders cannot independently verify the result. This project is trying to avoid both failures at the same time.

The trick is not “private voting” or “public auditing” alone. The trick is separating authorisation from ballot publication without losing verifiability.

Open with the contradiction. Frame the entire deck as resolving a three-way tension: privacy, auditability, and deployability.
The Core Problem

Online voting pulls in four directions at once

1

Eligibility

Only real, allowed participants should be able to vote.

2

Anonymity

No coordinator should be able to map a public ballot back to the voter.

3

Uniqueness

Each voting credential must be spendable once, with deterministic conflict resolution.

4

Auditability

Any observer should be able to recompute the tally from public evidence.

Common bad shortcut

  • Operator checks identity
  • Operator also sees the vote
  • Audit depends on trusting the server

What we actually need

  • Voter proves they may vote
  • Operator issues a credential without learning the final token
  • Ballot is public, but unlinkable
This slide explains why the problem is structurally hard, not just under-engineered.
Why Most Systems Fail

The awkward truth: normal web architecture is biased against private verifiable voting

Central database bias

Traditional apps want one backend to know identity, authorisation, current state, and final outcome. That is convenient for product teams and bad for ballot secrecy.

Authentication leakage

If the same identity that authenticates also publishes the vote, anonymity is gone. If the identity disappears too early, eligibility becomes hard to prove.

Audit theater

Many “auditable” systems still require trusting server logs, private tables, or privileged access. That is not public verification.

The Thesis

Make authorisation private, make ballots public, and make the bridge between them blind

The separating line that makes the system possible

Real identity

Voter proves they belong in the eligible set.

Blind issuance

Coordinator signs a blinded message, not the final token.

Local assembly

Voter unblinds and combines shares in the browser.

Ephemeral ballot

Vote is published from a fresh key, not the long-term identity.

Public audit

Anyone checks validity, uniqueness, and tally from relay data.

Blind signatures are the privacy hinge. They let a coordinator authorise a vote without learning the exact credential that will later be spent in public.

This is the core design claim. Everything else is support infrastructure around this separation.
Why Nostr

Nostr gives you a signed event fabric instead of a single authoritative server

What Nostr contributes

  • Public, signed event publication for rounds, ballots, and results
  • Private NIP-17 DM transport for issuance requests and responses, with retry for unacknowledged ticket delivery
  • Optional NIP-65 inbox/outbox hints for relay discovery when enabled
  • Relay diversity instead of one central database
  • Portable identities and signed announcements

Why that matters here

  • Auditors can read the same public state as coordinators
  • Static clients can participate without a proprietary backend
  • Coordinator rosters and per-round keys can be announced in public
  • The lead coordinator can push share-index assignments to sub-coordinators before issuance starts
  • Coordinator follower rows now expose per-ticket relay publish diagnostics for debugging delivery gaps
  • The default path currently prefers a tight curated relay set over broad hint expansion
  • The protocol remains inspectable by any third party

Threshold Model

Multiple coordinators make fraud harder and auditing more meaningful

What threshold issuance changes

  • The round defines the authorised coordinator roster
  • Each coordinator announces a per-round blind key
  • The voter gathers enough distinct shares to reach threshold T of N
  • The final ballot is valid only if enough authorised coordinators participated

What auditors gain

  • They can verify the ballot used the right roster for that round
  • They can reject shares from unauthorised coordinators
  • They can compare public tallies across coordinators
  • They can detect censorship or inflation by divergence

End-to-End Flow

Why the system is possible: every phase has a clear public/private boundary

Protocol phases

1. Publish round

Public on Nostr: question, voting ID, threshold, authorised coordinators.

2. Announce blind keys

Public on Nostr: per-round coordinator keys signed by stable identities.

3. Issue blind shares

Private over DMs: blinded request in, blind signature share out.

4. Publish ballots

Public on Nostr: ephemeral-key vote + anonymous proof material, without issuance-linking metadata.

5. Recompute tally

Anyone audits validity, duplicate rejection, and final totals.

Private

Eligibility proofing, blind issuance requests, blind signature responses, local secrets.

Public

Round metadata, coordinator roster, ballot stream, accepted tally logic, results. The public ballot must reveal enough to verify, but not enough to reconnect it to issuance.

Local only

Unblinding, token assembly, ephemeral ballot key generation, user-side backup and restore.

Auditing

The system is only credible if outsiders can reconstruct the result themselves

What an auditor reads

  • Round announcement and coordinator roster
  • Per-round blind-key announcements
  • Public ballot events
  • Final results from coordinators

What an auditor checks

  • Was each ballot bound to the correct round?
  • Did it meet threshold from authorised coordinators only?
  • Was the token spent more than once?
  • Does the public tally match the accepted valid ballots?

“Auditor” is not an administrator role. It is the proof that the protocol state is public enough to be challenged by anyone.

This connects directly to the new Auditor screen in the client: auditability is a protocol property, not just a dashboard feature.
Limits

This is possible, but only if the hard edges are treated as protocol work, not UI polish

Relay reality

Public relays are lossy, delayed, and uneven. The client must reconcile history, not trust live delivery alone.

Key custody

If coordinator signing keys sit unprotected in the browser, the privacy math is irrelevant. Local secret handling matters.

Canonical spend rules

“First valid spend wins” only works if every verifier computes “first” the same way from public data.

What this stack can do well

  • Publicly auditable rounds and ballots
  • Static client deployment
  • Coordinator-blind authorisation
  • Portable observer verification

What still needs discipline

  • Strong blind-signature implementation
  • Deterministic duplicate handling
  • Encrypted local key storage
  • Relay health and retry strategy

The current repo is already client-only and static-deployable, but live relay reliability is still the main operational weakness.
