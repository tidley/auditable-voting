Why auditable anonymous voting is hard, and why this stack can make it work

The usual failure mode is simple: either the operator can tell who voted for what, or outsiders cannot independently verify the result. This project is trying to avoid both failures at the same time.

The trick is not “private voting” or “public auditing” alone. The trick is separating authorisation from ballot publication without losing verifiability.

Open with the contradiction. Frame the entire deck as resolving a three-way tension: privacy, auditability, and deployability.
The Core Problem

Online voting pulls in four directions at once

1

Eligibility

Only real, allowed participants should be able to vote.

2

Anonymity

No coordinator should be able to map a public ballot back to the voter.

3

Uniqueness

Each voting credential must be spendable once, with deterministic conflict resolution.

4

Auditability

Any observer should be able to recompute the tally from public evidence.

Common bad shortcut

  • Operator checks identity
  • Operator also sees the vote
  • Audit depends on trusting the server

What we actually need

  • Voter proves they may vote
  • Operator issues a credential without learning the final token
  • Ballot is public, but unlinkable
This slide explains why the problem is structurally hard, not just under-engineered.
Why Most Systems Fail

The awkward truth: normal web architecture is biased against private verifiable voting

Central database bias

Traditional apps want one backend to know identity, authorisation, current state, and final outcome. That is convenient for product teams and bad for ballot secrecy.

Authentication leakage

If the same identity that authenticates also publishes the vote, anonymity is gone. If the identity disappears too early, eligibility becomes hard to prove.

Audit theater

Many “auditable” systems still require trusting server logs, private tables, or privileged access. That is not public verification.

Forget the voter

The Thesis

Make authorisation private, make ballots public, and make the bridge between them blind

The separating line that makes the system possible

Real identity

Voter proves they belong in the eligible set.

Blind issuance

Coordinator signs a blinded message, not the final token.

Local assembly

Voter unblinds and combines shares in the browser.

Ephemeral ballot

Vote is published from a fresh key, not the long-term identity.

Public audit

Anyone checks validity, uniqueness, and tally from relay data.

Blind signatures are the privacy hinge. They let a coordinator authorise a vote without learning the exact credential that will later be spent in public.

This is the core design claim. Everything else is support infrastructure around this separation.
Why Nostr

Nostr gives you a signed event fabric instead of a single authoritative server

What Nostr contributes

  • Public, signed event publication for rounds, ballots, and results
  • Coordinator-control carrier events for round proposals, commits, tally coordination, and recovery checkpoints
  • Private encrypted mailbox objects for issuance requests, ticket delivery, and ticket acknowledgements, with stable ids and relay-history backfill
  • In initial course-feedback mode, acknowledgements are diagnostic and valid ballot acceptance is the completion signal
  • Ballot events now carry stable `request_id` / `ticket_id` lineage so coordinator mapping and harness acceptance counts follow Rust truth
  • Ticket scheduler now surfaces first-send vs resend queue diagnostics with env-tunable concurrency and retry age for live reliability tuning
  • Observation recovery now tracks live vs backfill visibility and resend-recovered tickets in top-level harness output
  • Startup control diagnostics now capture publish payloads, precise-vs-broad startup reads, and relay write/read overlap before mailbox tuning
  • Single-coordinator runs now bypass MLS startup waits and use deterministic control readiness for round setup gating
  • Blind-key stalls are now classified as not-attempted vs publish/observe/apply failures with event and relay diagnostics
  • Questionnaire protocol scaffolding now exists for private response bundles, state transitions, and public aggregate result events
  • The coordinator questionnaire screen now follows staged tabs: Build, Audience, Publish, Responses, and Results, with explicit publish readiness checks
  • Accepted followers now receive active questionnaire ids (`open`/`published`) in coordinator roster DMs so voter questionnaire selection can auto-populate
  • Voter Vote now stays disabled until announced questionnaire ids are verified as publicly readable (`definition` + `open`/`published` state)
  • Questionnaire reads now use direct live subscriptions with one startup backfill plus one bounded retry, and expose per-voter discovery timing diagnostics for startup failures
  • Voter questionnaire responses now stay intact when the blind ballot credential or same-questionnaire definition refresh arrives
  • Course-feedback coordinator mode now bypasses legacy live-round, blind-key, and ticket queue gating when accepting questionnaire responses
  • Course-feedback harness runs now use batch-gated enrolment/submission waves (`LIVE_BATCH_SIZE`, default 5) with checkpointed progress instead of all-voter cold-start fanout
  • Coordinator questionnaire response observation now falls back to bounded kind-only reads plus local questionnaire-id filtering when tag-indexed relay reads are unreliable
  • Voter questionnaire submissions now spend an RSABSSA blind-signed credential from a fresh ephemeral response npub, with one accepted spend per credential
  • Voter mailbox reads now key by `request_id` with immutable per-request mailbox bindings, plus read/backfill mailbox consistency diagnostics
  • Voter questionnaire participation history is now stored locally and included in voter backup/restore bundles
  • Optional NIP-65 inbox/outbox hints for relay discovery when enabled
  • Relay diversity instead of one central database
  • Portable identities and signed announcements

Threshold Model

Multiple coordinators make fraud harder and auditing more meaningful

What threshold issuance changes

  • The round defines the authorised coordinator roster
  • Each coordinator announces a per-round blind key
  • The voter gathers enough distinct shares to reach threshold T of N
  • The final ballot is valid only if enough authorised coordinators participated

What auditors gain

  • They can verify the ballot used the right roster for that round
  • They can reject shares from unauthorised coordinators
  • They can compare public tallies across coordinators
  • They can detect censorship or inflation by divergence
Trust distribution

Coordinator A

Per-round blind key + share

Coordinator B

Per-round blind key + share

Voter

Combines enough shares locally

Auditor

Checks threshold and coordinator set from public state

The audience should come away understanding that multiple coordinators are not ornamentation; they change the trust model.
End-to-End Flow

Why the system is possible: every phase has a clear public/private boundary

Protocol phases

0. Coordinator control

Private over Nostr carriers: typed draft / proposal / commit events replayed in Rust/Wasm until round open is agreed.

1. Publish round

Public on Nostr: question, voting ID, threshold, and authorised coordinators, but only after coordinator control reaches agreement.

2. Announce blind keys

Public on Nostr: per-round coordinator keys signed by stable identities.

3. Issue blind shares

Private over mailbox objects: blinded request in, blind signature share out, acknowledgement back through the same mailbox lineage.

4. Publish ballots

Public rounds use an ephemeral-key vote. Questionnaires submit privately from a fresh ephemeral response npub with the unblinded credential.

5. Recompute tally

Anyone audits validity, duplicate rejection, and final totals.

Private

Coordinator control messages, eligibility proofing, blind issuance requests, blind signature responses, local secrets.

Public

Round metadata, coordinator roster, ballot stream, accepted tally logic, results. The public ballot must reveal enough to verify, but not enough to reconnect it to issuance.

Local only

Unblinding, token assembly, ephemeral ballot key generation, user-side backup and restore.

Auditing

The system is only credible if outsiders can reconstruct the result themselves

What an auditor reads

  • Round announcement and coordinator roster
  • Per-round blind-key announcements
  • Public ballot events
  • Final results from coordinators

What an auditor checks

  • Was each ballot bound to the correct round?
  • Did it meet threshold from authorised coordinators only?
  • Was the token spent more than once?
  • Does the public tally match the accepted valid ballots?

“Auditor” is not an administrator role. It is the proof that the protocol state is public enough to be challenged by anyone.

This connects directly to the new Auditor screen in the client: auditability is a protocol property, not just a dashboard feature.
Limits

This is possible, but only if the hard edges are treated as protocol work, not UI polish

Relay reality

Public relays are lossy, delayed, and uneven. The client must reconcile history, not trust live delivery alone.

Key custody

If coordinator signing keys sit unprotected in the browser, the privacy math is irrelevant. Local secret handling matters.

Canonical spend rules

“First valid spend wins” only works if every verifier computes “first” the same way from public data.

What this stack can do well

  • Publicly auditable rounds and ballots
  • Static client deployment
  • Optional downloadable Rust worker helper with shareable autoconfigured launchers and Advanced CLI path (outbound-only relay mode)
  • Coordinator-offline blind-request routing via cached worker hints
  • Worker-issued blind credentials now carry the questionnaire definition needed to render ballots offline
  • Coordinator-blind authorisation
  • Portable observer verification

What still needs discipline

  • Strong blind-signature implementation
  • Deterministic duplicate handling
  • Encrypted local key storage
  • Relay health and retry strategy

The current repo is already client-only and static-deployable. Small local gates are better but still not fully boring: `1 / 1 / 1` and `1 / 2 / 2` are good, while repeated larger public-relay committee runs are still not signed off. The live browser supervisory path now uses the OpenMLS bootstrap/join flow, blocks the first round until sub-coordinator welcome acknowledgement is visible after the non-lead has completed an initial coordinator-control sync, gives coordinator-control and ticket/ack traffic a slightly wider primary relay subset than ordinary DMs, queues DM publishes per sender-recipient conversation, serialises one coordinator's mailbox ticket publishes through a sender-scoped queue, keeps one deterministic mailbox anchor relay while rotating secondary relays by recipient, temporarily cools down relays that return rate-limit or policy failures, adds random 0-30s human-style delays before automatic follow/request/ticket/ack sends, preserves prior ticket attempt ids so later acknowledgements are not lost when retries replace the latest tracked response, keeps shard-request ids stable end to end, and surfaces coordinator runtime-readiness phases plus protocol-layer failure classes (`startup`, `dm_pipeline`, `mixed`) with coordinator/voter readiness snapshots in debug evidence. The current large-run blocker is still unstable enough to present as more than one failure shape.

Takeaway

Why this works: the stack aligns the trust boundaries with the protocol boundaries

Nostr handles shared signed public state. Blind signatures handle private unlinkable authorisation. Ephemeral keys handle public ballot separation. Auditors handle independent recomputation.

Public
